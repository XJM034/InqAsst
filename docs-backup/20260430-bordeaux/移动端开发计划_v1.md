本文档用于固化 InqAsst 第一版移动端开发方案，作为后续工程初始化、页面开发、接口联调和验收的统一依据。

## 2. 第一版目标

第一版目标不是做完整教务系统，而是先交付一个可运行的移动端业务闭环：

- 老师可通过手机号登录，查看当天课程并完成点名。
- 管理员可通过移动端查看点名总控、未到学生，并完成现场应急处理。
- 页面样式基于 `design.pen` 的设计变量与组件结构落地，不采用脱离设计源的随意实现。
- 前端先完成可演示版本，再接入真实认证、真实课程数据和真实点名接口。

第一版上线后，系统应优先解决“管理端尽早发现未到学生”的问题，而不是扩展更多非核心功能。

## 2.1 当前实现同步

截至当前版本，仓库中的移动端前端已落地并可构建，且以下交互已经实现：

- 单角色手机号直达登录；`/role-select` 仅保留兼容重定向。
- 老师端：首页、点名页、只读学生名单、我的页与退出确认。
- 管理端：首页、点名总控、班级名单、未到学生、老师设置、课程与班级设置、时间设置、我的页。
- 管理端点名能力：
  - 班级名单页单个学生状态切换
  - 未到学生列表直接切换学生状态
  - 班级名单页支持先选择一批学生，再批量修改为已到、请假或未到

## 4. 产品决策

### 4.1 应用形态

第一版采用一个应用按角色分流的方式，而不是老师端和管理端拆成两个独立前端项目。

固定路由结构：

- `/login`
- `/teacher/*`
- `/admin/*`

规则如下：

- 单角色账号登录后直接进入对应首页。
- 一个手机号只允许对应一个角色入口。
- 登录后按手机号归属直接进入对应首页，不再经过角色选择页。

### 4.2 交付顺序

交付节奏固定为：

1. 先完成前端页面、交互、状态流和 mock 数据。
2. 再对接真实短信验证码、真实课程数据和真实点名接口。

虽然先做前端，但第一版就必须把领域模型、接口契约和权限结构按真实业务设计清楚，避免后期重构。

### 4.3 部署方式

部署目标为自托管 Docker。

第一版前端工程必须满足以下要求：

- 本地开发可直接启动
- 构建产物可 Docker 化
- 环境变量可在运行时注入
- 后续接入内网或自有数据库时无需更换前端框架

## 5. 技术选型

### 5.1 核心技术栈

- 框架：Next.js App Router
- 语言：TypeScript
- 样式：Tailwind CSS
- 组件库：shadcn/ui
- 图标库：Lucide
- 测试：
  - Vitest
  - Testing Library
  - Playwright

### 5.2 选择原因

选择 Next.js App Router 的原因：

- 可用一个 React/TypeScript 工程同时承载页面、服务端接口和未来的登录态处理。
- 适合自托管 Docker 部署。
- 支持 Server Components、Route Handlers，便于未来接正式后端时做分层。

选择 shadcn/ui 的原因：

- 组件开源、可复制到本项目内部，适合做内部业务系统的长期维护。
- 与 Tailwind CSS、CSS variables 配合自然，便于承接 Pencil 的设计变量。
- 适合作为基础交互层，再往上封装业务组件。

## 6. 设计到代码实施方案

### 6.1 总体原则

前端页面实现必须围绕 `design.pen` 进行，不采用“看着原型大概写一个类似页面”的方式。

实现链路固定为：

1. 保留 `design.pen` 在仓库根目录。
2. 读取 `design.pen` 中的 variables。
3. 将 variables 同步到前端 `app/globals.css`。
4. 使用 shadcn/ui 基础组件承接交互。
5. 基于业务需求封装应用组件。
6. 再组合成页面。

### 6.2 Pencil Variables 同步方案

按 Pencil 官方文档的 `CSS Variables ↔ Pencil Variables` 方式处理。

同步目标分为两层：

- shadcn 语义变量
- Pencil 业务变量

必须保留并使用的语义变量包括：

- `--background`
- `--foreground`
- `--card`
- `--card-foreground`
- `--border`
- `--input`
- `--primary`
- `--primary-foreground`

- `--muted`
- `--ring`

### 6.3 主题策略

`design.pen` 中包含 Light 和 Dark 主题定义。

本期策略：

### 6.4 icon 策略

## 7. shadcn/ui 配置方案

### 7.1 components.json 约定

项目初始化时创建 `components.json`，至少满足以下配置方向：

- `rsc: true`
- `tsx: true`
- `tailwind.css: "app/globals.css"`
- `tailwind.cssVariables: true`
- `baseColor: "neutral"`
- `iconLibrary: "lucide"`

### 7.2 目录分层

组件分为两层：

- `components/ui/*`
- `components/app/*`

其中：

- `components/ui/*` 直接承接 shadcn/ui 基础组件。
- `components/app/*` 承接业务组件，不允许把业务逻辑堆进 `components/ui/*`。

### 7.3 必选基础组件

初始化时至少纳入以下基础组件：

- `button`
- `card`
- `dialog`
- `drawer`
- `input`
- `input-otp`
- `label`
- `badge`
- `tabs`
- `select`
- `separator`
- `scroll-area`
- `sheet`

若后续发现管理端页面对某些组件有强依赖，可再增补，但第一版不追求“先把全套组件装完”。

第一版虽然用 mock 数据，但接口形状必须直接对齐未来正式后端。

建议 service 接口如下：

- `requestOtp(phone)`
- `verifyOtp(phone, code)`
- `getCurrentUser()`
- `switchRole(role)`
- `getTeacherHome(date)`
- `getAttendanceSession(sessionId)`
- `submitAttendance(payload)`
- `getAdminSummary(date)`
- `getAdminControlBoard(date)`
- `getUnarrivedStudents(date)`
- `updateStudentAttendanceStatus(payload)`
- `updateEmergencyTeacherPhone(payload)`
- `createSubstituteTeacher(payload)`
- `updateEmergencyCourseTime(payload)`

### 12.1 服务层约束

- 页面不得直接 import mock JSON。
- 页面统一通过 service 层读取数据。
- 后续替换正式接口时，不改变页面调用方式，只替换 service 实现。

## 13. 状态管理方案

第一版采用轻量状态管理，不引入全局状态库。

方案如下：

- 首屏数据由服务端组件获取
- 页面内部交互状态由客户端组件管理
- 会话与角色信息通过统一 auth/session 模块维护

本期不引入：

- Redux
- Zustand
- MobX

如果后续真实联调阶段状态复杂度明显上升，再单独评估是否增加全局状态层。

## 17. 风险与默认决策

### 17.1 已确定默认决策

- 第一版只做移动端
- 一个应用按角色分流
- 前端先行，后端后接
- 最终认证方式是短信验证码
- 当前阶段允许使用 mock OTP 占位

### 17.2 主要风险

- 如果 mock 数据结构与真实后端字段偏离，后续联调成本会快速上升
- 如果页面开发阶段跳过 token 层，直接用零散样式实现，后续统一视觉会很痛苦
- 如果业务规则写在页面里，而不是沉淀到 domain/service 层，后续接真实接口会变得脆弱

### 17.3 风险应对

- 先定义领域模型和接口契约，再写页面
- 先完成 token 同步，再搭 UI
- 页面只消费 service，不直接碰 mock 数据

## 18. 下一步执行建议

建议从以下顺序开始执行：

1. 初始化 Next.js、Tailwind、shadcn/ui。
2. 从 `design.pen` 导出并接入 CSS variables。
3. 建立 `components/ui` 和 `components/app` 分层。
4. 先完成登录页、老师首页和点名页。
5. 再补管理首页、点名总控和未到学生列表。
6. 最后补管理端应急设置和学生页面壳子。

至此，第一版移动端开发计划定稿，可作为工程实施基线。
