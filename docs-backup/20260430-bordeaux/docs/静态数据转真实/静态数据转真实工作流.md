# 静态数据转真实工作流
日期：2026-04-18

适用范围：任何“页面先用静态数据跑通，后续再切换为真实后端接口”的功能。

当前范围备注：学生的批量导入功能因业务方案尚未设计完成，当前先不开发，只保留页面盘点、缺口记录和风险说明，不纳入本轮真实化开发与联动改造范围。
补充备注：如果某个页面已经没有静态数据、Mock、假链路问题，即使仍保留 legacy 路由或兼容壳，也可以先不处理，不作为当前阶段阻塞项。

## 目标

把前端页面从静态数据、Mock、H2、假接口、假仓储、临时兼容分支，切换为真实接口 + 真实数据库支撑的正式实现，并在改完后通过测试闭环验证。

## 核心原则

1. 前端展示的数据必须来自真实后端接口。
2. 后端接口必须有真实数据库支撑，不能停留在内存、Mock、假表结构或临时脚本层。
3. 一旦真实链路可用，前端原有静态数据、Mock、H2、假实现必须移除，不能长期新旧并存。
4. 文档、接口、数据库、前端调用、测试口径必须一致。
5. 改完功能走 `tests` 测试，测试不成功继续改，直到成功为止。

## 功能思考链路

做“静态数据转真实”时，不能只按页面症状修问题，而要先按“这个功能真正做了什么”来拆链路。

优先按下面顺序思考：

1. 这个功能的唯一真实数据源是谁。
2. 这个功能的写操作，实际会改变什么状态。
3. 哪些读接口、读服务、页面摘要、列表、统计、入口判断，会依赖这份状态。
4. 前后端是不是都在读取同一套真实结果，而不是一部分读新口径、一部分还在读旧口径。
5. 测试是不是在验证“操作前后数据是否联动”，而不只是验证页面能显示。

要求：

- 不要只看到“总数没变”“文案没刷新”“列表没同步”就停在表层修补。
- 要先确认这是不是同一个功能状态没有贯通到所有读取方。
- 一个功能只要存在新增、移除、切换、启停、同步等写操作，就默认要检查所有依赖它的读链路。
- 不能等用户逐个指出症状后再逐个补洞，要主动完成一次全链路联动核对。

## 数据库核验原则

凡是涉及数据库能力判断、字段存储位置、表职责划分、是否需要改库、是否已有真实支撑等问题，必须先核验真实数据库结构，不能凭代码语义、接口命名、实体字段名或经验推断。

必须优先核验：

1. schema / 建表脚本
2. migration / 增量变更脚本
3. entity 映射
4. repository 查询与写入方式
5. 如有条件，再核验当前实际库表结构

要求：

- 先确认“数据库现在已经有什么”，再讨论“应该怎么改”。
- 必须明确区分“现状事实”和“设计建议”，不能混说。
- 不能因为后端 service 里有某段逻辑，就默认数据库已经有对应字段或表支撑。
- 不能因为 entity 里有某个字段，就默认实际数据库已经完成迁移。
- 涉及是否加字段、是否加表、标记怎么存、唯一键和约束是否足够时，必须先完成数据库核验，之后才能进入实现判断。

## 推荐分层

### 后端推荐分层

按下面顺序补齐真实支撑：

1. 数据库迁移 / schema
2. entity / repository
3. service
4. controller
5. request / response DTO
6. test

要求：

- 先补数据库结构，再补接口代码。
- 每个新增或改造接口，都要能追到具体表、字段、约束、索引或查询写入逻辑。
- 不允许只做 controller 返回演示数据，数据库层没有真实落点。

### 前端推荐分层

按下面顺序接入真实链路：

1. schema / DTO 类型
2. API client
3. adapter
4. domain type
5. page / component
6. test

要求：

- 页面不要直接消费后端原始 DTO。
- 真实请求、错误处理、响应解析，集中放在 API client。
- DTO 到前端领域模型或页面模型的转换，集中放在 adapter。
- 页面只依赖已经整理好的 domain 数据，不直接拼装假字段补洞。

## 标准工作流

### 1. 先做功能级联动核对

在盘点静态入口之前，先把功能本身的状态流转画清楚：

1. 功能入口是什么
2. 写操作有哪些
3. 写操作修改后的真实状态存在哪里
4. 哪些页面、接口、统计、摘要、入口判断会读取这份状态
5. 当前代码里是否存在多个读取口径

输出结果：

- 一份“功能写入点清单”
- 一份“依赖该功能状态的读取方清单”
- 一份“新旧口径不一致风险清单”

### 2. 盘点静态入口

先找出当前功能还在走哪些“假链路”：

- 页面内写死常量
- Mock 文件
- 前端本地假服务
- H2 或其他仅用于占位的假数据库
- 临时 adapter
- 临时兼容分支
- 任何不是来自真实数据库的数据来源

输出结果：

- 一份“静态/假实现入口清单”
- 一份“需要替换成真实链路的调用清单”

### 3. 建立字段到数据库的追踪关系

把链路拆完整：

1. 前端页面需要哪些字段
2. 对应后端要提供哪些接口
3. 每个接口对应哪些表、字段、关系、约束
4. 哪些地方缺表、缺字段、缺索引、缺查询、缺写入逻辑
5. schema、migration、entity、repository 是否彼此一致

要求：

- 每个前端字段都能追到接口。
- 每个接口都能追到数据库。
- 每个数据库结论都要能追到 schema / migration / entity / repository 中至少一处真实依据。
- 如果追不到，说明还不能进入前端切换阶段。

### 4. 先补后端真实支撑

如果后端还没有真实能力，先补后端，不要先在前端做临时拼接。

至少补齐：

- migration / schema
- entity / repository
- service 编排与校验
- controller / DTO
- 后端测试

要求：

- 接口返回值覆盖前端真实需要，不返回演示性质字段。
- 写接口要有真实落库逻辑，读接口要有真实查询逻辑。
- 所有接口都要有数据库支撑，不能靠内存态、临时 Map、假仓储顶住。
- 同一个功能相关的读接口，要统一读取同一份真实状态，不允许各算各的。

### 5. 定义接口契约与前端接入方式

在前端正式切换前，把接口契约和接入分层定清楚：

- schema 定义后端响应结构
- client 负责请求与错误归一
- adapter 负责 DTO 转页面可用结构
- domain type 负责页面内部稳定类型

要求：

- 不要让页面直接依赖后端原始字段命名。
- 不要把字段兼容、脏数据兜底、假默认值散落在多个页面组件里。
- 契约变化时，优先改 schema / adapter，不要让页面到处跟着改。

### 6. 前端切到真实链路

后端能力和契约稳定后，再改前端：

- 接入真实 API client
- 替换静态数据来源
- 补齐 adapter 映射
- 调整页面加载、空态、异常态
- 删除依赖假数据才能工作的逻辑

要求：

- 页面最终只保留真实链路。
- 能删掉的临时兼容逻辑就删掉，不保留长期过渡层。
- 同一个功能相关的页面、摘要、数量、状态标签、入口跳转，要共享同一份最新真实数据，不要各自维护一套旧状态。

### 7. 清理所有假实现

真实链路接通后，集中清理：

- Mock 数据文件
- H2 假数据库方案
- 假服务
- 假接口路由
- 页面写死数据
- 只服务于静态演示的 adapter 或兼容层

要求：

- 清理要彻底。
- 不保留“以后可能还会用”的无主假逻辑。
- 如果测试必须保留桩数据，只能留在测试目录，并且命名、用途明确。

### 8. 对齐文档

同步更新：

- 接口说明
- 数据来源说明
- 联调说明
- 已删除的 Mock / H2 / 静态链路说明

要求：

- 文档描述必须和代码现状一致。
- 不能出现“文档说是真实接口，代码实际还在读假数据”。

### 9. 测试闭环

改完后必须执行测试，并形成循环：

1. 先跑最小范围单测 / service 测试
2. 再跑接口契约或 controller 测试
3. 再跑联调或端到端验证
4. 失败就继续修，修完重新跑

联动测试要求：

- 至少覆盖一次“执行写操作前”和“执行写操作后”的数据对比。
- 要验证的不只是列表本身，还包括数量、摘要、状态展示、入口判断、依赖该状态的其它页面或接口。
- 如果一个功能存在多个读取方，测试要证明它们读取的是同一套真实结果。

执行要求：

`改完功能走 tests 测试，测试不成功继续改，直到成功为止。`

## 完成判定

满足以下条件，才算“静态数据转真实”完成：

1. 页面不再依赖静态数据、Mock、H2、假接口或假数据库。
2. 前端所有展示字段都来自真实接口。
3. 后端所有相关接口都有真实数据库支撑。
4. 前后端链路已经按 schema / client / adapter / domain / page 分层落地。
5. 假实现已清理完成，没有遗留无主冗余代码。
6. 文档、接口、数据库、前端实现已经对齐。
7. 同一个功能相关的读取方已经完成联动核对，不存在一部分读新口径、一部分读旧口径。
8. 相关测试已执行并通过。

## 禁止事项

1. 用真实接口包一层假数据再返回给前端。
2. 主链路已经接通后，长期保留旧的 Mock / H2 方案。
3. 为了让页面先显示，加入无法追踪来源的硬编码字段。
4. 把 DTO 转换、异常兼容、字段兜底散落在页面组件里。
5. 测试失败后直接结束任务，或带着失败状态交付。
6. 把具体业务页面名、具体业务实体名写进通用工作流，污染后续复用。

## 交付建议

每次按这个工作流推进，至少产出：

- 一份真实链路改造说明
- 一份已删除假实现清单
- 一份测试结果说明

这样后续其它功能做“静态数据转真实”时，可以直接复用这套方法，而不是重新摸索流程。
