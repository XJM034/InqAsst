## 问题陈述

K12 课后延时教育公司面临的核心痛点：
1. **走班制下师生找不到上课地点** - 外部老师不熟悉校园,学生在多个校区间流动
2. **考勤管理分散低效** - 老师现场点名,管理员需要汇总统计,缺乏实时可见性
3. **信息碎片化导致管理混乱** - 管理员面对迷路学生/未带工牌老师,需要在碎片化线索下快速定位

本次设计基于甲方最新反馈,回到双角色架构(老师端 + 管理端),聚焦 P0 核心功能:查询和考勤。

## 核心设计理念

- **极简信息架构** - 老师端是"智能行程单",管理端是"全能搜索与管控台"
- **高容错率** - 支持拼音、首字母、同音字模糊搜索,应对信息碎片化场景
- **上帝视角** - 管理端可全局查询学生/老师/课程,快速"破案"

## 约束条件

1. **角色架构**: 本期只做双角色(老师 + 管理员),v2 的助教端、通知系统、推送系统全部延后
2. **校区权限**: 管理员单校区绑定不可切换;老师可跨校区,按天自动显示当天所在校区
3. **双端协同**: 老师端与管理端保留移动端; 管理端补充 Web 端运营台,承担结构化导出、财务结算与异常处理
4. **技术边界**: 设计文档只描述需求和交互,不涉及具体技术实现方案

## 前提条件

1. 本期回到双角色架构,v2 的三角色架构(助教端、通知系统、推送系统)全部延后
2. 搜索功能需要支持同音字匹配(如"李嘉树"匹配"李佳澍"),后端需将姓名转换为拼音存储
3. 考勤时间权限明确:老师端在课程实例时间内可修改考勤;课后默认只读,如需修改需发起申请解锁;管理端课中不直接改最终考勤,只发起协作请求
4. 学生名单布局从"一行一人"优化为"网格卡片布局(每行 3 人)",提高空间利用率
5. 管理端移动端保留"复制到剪贴板"型导出,用于微信群即时催促;同时新增 Web 端结构化导出中心,服务财务统计与历史查询
6. 运营结果与财务口径分离:业务侧实时更新事实状态,财务侧以每月 26 号为待结算点,人工确认后形成当期结算

## 考虑的方案

### 方案 A: 最小化调整(基于 v1.7 微调)

**概述**: 在现有 v1.7 架构基础上进行局部优化,保持整体结构不变,只调整具体交互细节。

**核心变更**:
- 搜索功能增加同音字匹配能力
- 学生名单布局从列表改为网格(每行 2-3 人)
- 管理端考勤增加两个导出功能(复制到剪贴板)
- 删除学生个人特征标签

**优点**:
- 开发工作量最小,风险可控
- 与现有原型图差异小,易于理解和验收
- 可快速交付,满足甲方时间要求

**缺点**:
- 未从根本上优化信息架构,仍有改进空间
- 部分交互细节(如导出格式)需要在开发中细化

**工作量**: S (小型调整,预计 1-2 周开发周期)

**风险**: 低

**复用**: 完全基于现有 v1.7 设计,复用所有页面结构和组件

### 方案 B: 结构化重构(优化信息架构)

**概述**: 在满足甲方需求的同时,对信息架构进行系统性优化,提升整体用户体验。

**核心变更**:
- 管理端考勤页面重新设计:统计概览 + 班级列表 + 学生名单三层结构
- 管理端"未到学生列表"补充学生状态总览:已到 / 请假 / 未到 + 当日行课人数提示
- 老师端考勤增加快捷筛选(全部/已到/未到)和倒计时提醒
- 搜索结果优先级调整:学生 > 老师 > 课程
- 导出功能标准化:统一格式,支持微信群直接粘贴

**优点**:
- 信息层级更清晰,符合管理员和老师的实际工作流
- 考勤统计可见性更强,管理员可快速掌握全局状态
- 导出功能更实用,减少管理员手工整理工作

**缺点**:
- 需要重新设计部分页面结构,开发工作量增加
- 与现有原型图差异较大,需要重新评审

**工作量**: M (中型重构,预计 2-3 周开发周期)

**风险**: 中

**复用**: 复用现有组件和交互模式,但页面结构需要调整

## 推荐方案

**推荐方案: 方案 A 基础上对管理端考勤做定向扩展**

**理由**:
1. **时间优先** - 乙方接单项目,甲方已经过多轮需求对齐,现在需要快速交付而非大幅优化
2. **局部扩张而非整体翻修** - 老师端与管理端移动端仍沿用现有结构,但管理端考勤已超出按钮级微调,需定向扩展为"移动端现场督促 + Web 端运营台"
3. **规则先行** - 本轮最重要的不只是界面,而是老师端、管理端、主管、财务之间的协作边界,以及月度结算与更正链路
4. **可迭代** - 第一版先做实时运营面板、结构化导出、待更正清单与轻量异常闭环;补课/调课/非常规时间规则留待后续扩展

因此,本期整体交付节奏仍按方案 A 控制,但管理端考勤模块按"定向扩展"落地。

## 详细设计规范

### 一、全局规范

#### 1.1 角色与权限

**角色定义**:
- **老师端(Teacher)**: 外部上课老师,查看自己的课程行程,现场点名
- **管理端(Admin)**: 校区管理老师,全局搜索查询,考勤总控

**校区权限规则**:
- 管理员登录后只能看到自己所属校区的数据,不可切换校区
- 老师可能在不同校区上课,系统根据当天课程自动显示对应校区信息
- 所有页面默认显示当天(周一至周五)信息

#### 1.2 视觉规范

**地点格式标准**: 统一采用层级化文本 `楼栋 > 楼层 > 教室号`
- 示例: **综合楼 A 栋 > 3 层 > 3年级 2 班**
- 字号需醒目,作为视觉焦点

**五色分类系统**: 课程类别采用颜色标识
- 科创(蓝色系) / 人文(青色系) / 体育(橙色系) / 劳作(绿色系) / 艺术(紫色系)
- 用于课程卡片背景色、类别图标等视觉元素

**标签系统**: 删除学生个人特征标签(如"调皮""敏感"等),保留课程类别标签

#### 1.3 底部导航

- **老师端**: 首页 / 考勤 / 我的
- **管理端**: 首页 / 考勤 / 我的
- 底部导航使用统一的悬浮 pill 结构,`tabBarPill` 视觉上贴页面底边,老师端和管理端保持一致

### 二、老师端设计

#### 2.1 首页(Teacher_Home) - P0 功能

**定位**: 智能行程单,系统根据账号直接告知"何时、去哪、哪个校区"

**页面结构**:

**A. 顶部导航区**
- 左侧: 用户头像 + 欢迎语(如"李老师,下午好")
- 右侧: 周历切换器(周一至周五横向排列)
  - 默认选中"今天"
  - 有排课的日期下方显示蓝色圆点标记
  - 点击不同日期,下方任务卡片内容刷新

**B. 核心任务卡片**

若当日有课,显示任务卡片(支持多张堆叠):

1. **卡片眉部**: 通栏深色标签,醒目显示 **📍 [校区名称]** (如:成华校区)
2. **卡片主体**:
   - 背景色根据课程类别显示淡淡的主题色
   - 第一行: **时间**(16:30 - 17:30,特大字号加粗) + **课程名称**(无人机飞行 A 班) + 类别图标
   - 第二行(视觉焦点): 独立色块容器,显示 **楼栋 > 楼层 > 教室**,右侧预留导航图标按钮
3. **卡片底部**:
   - ~~删除"查看班级名单"按钮~~ (本期不提供)
   - 保留"联系管理老师"按钮(一键拨号)

**C. 空状态**
- 若选中日期无课,显示插画及文案:"该日无教学安排"

#### 2.2 考勤页(Teacher_Attendance) - P0 功能

**定位**: 老师现场点名工具,快速完成学生考勤

**页面结构**:

**A. 课程列表**
- 显示当日所有课程,按时间排序
- 每个课程卡片显示:时间段 + 课程名称 + 地点 + 点名状态
- 点名状态:
  - 课前:显示"即将开始"
  - 上课时间段内:显示"开始点名"按钮
  - 课后:显示"已完成"或"查看详情"

**B. 点名页**

进入点名页后:

1. **顶部倒计时提醒条**:
   - 显示"请在 X 分钟内完成点名并提交"
   - 上课后开始倒计时,5 分钟内为正常提示(蓝色/灰色)
   - 超过 5 分钟后提示条变红色警告
   - 不阻断提交,只是视觉提醒

2. **快捷筛选与操作按钮**:
   - 左侧筛选按钮:全部 / 已到 / 未到
   - 右侧快捷操作:"☑ 全选已到"按钮
   - 默认选中"全部"
   - 点击切换筛选,下方学生列表实时刷新
   - 点击"全选已到"一键将所有学生标记为"已到"

3. **学生名单(网格卡片布局)**:
   - 每行显示 3 个学生卡片
   - 每个卡片显示:姓名 + 行政班级
   - 删除头像和学生个人特征标签
   - 考勤状态:未到(默认) / 已到
   - 点击卡片切换状态(未到 ↔ 已到)

4. **底部统计与提交**:
   - 实时统计:应到 X 人 / 已到 Y 人 / 未到 Z 人
   - "提交考勤"按钮

**C. 提交弹窗(三态逻辑)**

根据点名结果显示不同弹窗:

1. **未点名阻断态**(未点名人数 > 0):
   - 标题:"还有 X 名学生未完成点名"
   - 文案:"请确认所有学生考勤状态后再提交"
   - 按钮:仅"继续点名"(关闭弹窗)

2. **全勤确认态**(未点名人数 = 0 且未到人数 = 0):
   - 标题:"全员到齐"
   - 文案:"本节课应到 X 人,已全部到齐"
   - 按钮:"确认提交" + "返回修改"

3. **缺勤确认态**(未点名人数 = 0 且未到人数 > 0):
   - 标题:"确认提交考勤"
   - 文案:"本节课应到 X 人,已到 Y 人"
   - 未到学生列表:只显示"未到(Z人)"列表,不分组
   - 按钮:"确认提交" + "返回修改"

**时间与协作规则**:
- 老师在课程实例时间内可正常修改和提交考勤
- 课后老师端默认只读;如需修改,需发起"课后修改申请",填写结构化原因并可补充备注
- 管理端移动端和 Web 端都可审批课后修改;审批通过后仅按申请对象临时解锁,开放 10 分钟,超时需重新申请
- 课中管理端不直接改最终考勤,只发起协作请求(如"请确认请假""请核实是否已到场")
- 只要该次考勤中存在待处理协作请求,老师不能最终提交
- 老师处理协作请求时必须明确给出结果:接受并修改 / 拒绝并说明原因 / 暂无法确认
- 选择"暂无法确认"后,该请求进入高优先级异常队列,且仍继续阻止最终提交

#### 2.3 我的页(Teacher_Profile)

- 用户信息:头像、姓名、所属机构
- 设置入口:修改密码、退出登录等
- 删除标签管理相关功能

### 三、管理端设计

#### 3.1 首页(Admin_Home) - P0 功能

**定位**: 全能搜索与管控台,基于"上帝视角"快速定位学生/老师/课程

**页面结构**:

**A. 超级搜索框**

1. **搜索输入框**:
   - Placeholder:`搜学生、老师姓名、课程...`
   - 支持输入:汉字、拼音全拼、拼音首字母
   - **同音字匹配**:输入"李嘉树"能匹配到"李佳澍"(拼音相同)

2. **即时联想预览列表**:
   - 输入过程中触发即时联想
   - 键盘上方弹出半屏列表
   - 同时匹配学生库、教师库、课程库
   - **默认优先显示学生结果**(大多数场景是搜索学生)
   - 列表项带身份标签:
     - `[学生]` **李景行** —— 5年级 3 班
     - `[老师]` **王力** —— 无人机飞行课程
     - `[课程]` **乐高机器人 A 班** —— 16:30 / 科技楼 205

**B. 级联筛选器**(辅助缩小"找学生"的范围)

1. **第一排:课程类别**
   - `全部` `体育` `科创` `艺术` `劳作` `人文`

2. **第二排:年级**
   - `一年级` `二年级` `三年级` ... `六年级`

3. **第三排:行政班级**(动态显示)
   - 默认状态:隐藏或置灰
   - 激活逻辑:当用户在第二排选中某个年级后,第三排点亮并显示该年级下的班级

**C. 结果详情卡片**

根据选中对象类型(学生/老师/课程),动态渲染三种卡片:

**布局一:学生卡片**
- 左侧:学生照片(大图,点击可放大),若无照片显示性别图标
- 右侧上部(身份确权):
  - 姓名 + 行政班级
  - 校验信息:班主任姓名(用于口头核对:"你班主任姓什么?")
  - ~~删除生日等其他校验信息~~
- 右侧下部(核心指引):
  - 课程名称(带颜色标签)
  - 地点:**楼栋 > 楼层 > 教室**(醒目显示,红色加粗大字号)

**布局二:老师卡片**
- 左侧:教师照片或通用图标
- 右侧上部:
  - 姓名 + 身份(某某机构签约教师)
  - 电话图标(点击直接拨号)
- 右侧下部:
  - 当前状态:`正在上课` 或 `即将开始`
  - 课程名称
  - 地点:**楼栋 > 楼层 > 教室**

**布局三:课程卡片**
- 左侧:类别图标(齿轮/画板/足球等),背景为该类别标准色
- 右侧上部:
  - 课程名称 + 时间段
  - 班级人数:应到 X 人(点击弹出学生名单,用于点名/核对)
- 右侧下部:
  - 授课教师姓名(带电话图标)
  - 地点:**楼栋 > 楼层 > 教室**

#### 3.2 名单页(Admin_Roster)

**说明**: 本期移除名单页入口,管理端底部导航调整为"首页/考勤/我的"。学生/课程查询通过首页搜索完成,考勤管理通过考勤页完成。

#### 3.3 考勤页(Admin_Attendance) - P0 功能

**定位**: 考勤总控台(移动端现场督促视图),管理员掌握全局进度、催促老师处理协作请求,并快速查看未到学生;复杂运营、结构化导出与财务结算在 Web 端完成

**页面结构**:

**A. 顶部统计概览**
- 班级列表 Tab:显示 已完成点名 X 班 / 未完成点名 Y 班
- 未到学生列表 Tab:显示 已到 X 人 / 请假 Y 人 / 未到 Z 人
- 统计卡片下方补充文案:`当日共 XXX 名学生行课`
- 顶部同步显示当前时间与实时进度提示,增强现场督促的紧迫感
- 醒目展示,一目了然

**B. 导出功能区**

两个导出按钮(复制到剪贴板,位于 Tab 切换上方,在班级列表与未到学生列表两种状态下都显示):

1. **导出未完成点名班级信息**:
   - 格式示例:
     ```
     【未完成点名班级】2026-03-23
     1. 乐高机器人 A 班 - 王力老师
     2. 足球 U8 班 - 张教练
     ```
   - 用途:复制后粘贴到微信群,催促老师完成点名

2. **导出未到学生信息**:
   - 格式示例(按行政班级分组):
     ```
     【未到学生名单】2026-03-23

     【5年级3班】
     - 李景行 | 乐高机器人 A 班 | 科技楼 A 栋 > 2 层 > 205

     【4年级8班】
     - 王小明 | 足球 U8 班 | 北操场 3 号区
     ```
   - 用途:复制后粘贴到班主任群,让班主任回复未到情况

**补充说明**:
- 移动端导出继续服务即时沟通,不承担财务统计
- 结构化导出统一放在 Web 端考勤管理台中完成

**C. 搜索框**
- 位置:班级列表 Tab 下位于导出按钮下方;未到学生列表 Tab 下位于 Tab 切换下方
- Placeholder:`搜索学生姓名...`
- 功能:支持模糊搜索学生姓名,快速定位到该学生所在班级
- 用途:班主任回复"XXX请假了"后,管理员可直接搜索学生姓名进入对应班级修改状态

**D. 双 Tab 切换**

考勤页内容区分为两个 Tab:

**Tab 1: 班级列表**

显示当日所有课程班级:
- 每个班级卡片显示:班级名称 + 老师姓名 + 点名状态
- 默认排序:未完成考勤的班级优先展示,已完成考勤的班级排在后面

#### 3.4 Web 考勤管理台(Admin_Attendance_Web)

**定位**: 管理端统一运营台,承接实时运营、结构化导出、异常处理与财务结算

**信息架构**:
- **实时运营面板**: 轻量 dashboard + 动作区混合布局,顶部显示当前时间、提交率、超时班级数、未到学生数;中下部按动作优先展示未提交班级、协作待回执、课后申请、主管未响应等任务
- **结构化导出中心**: 第一版采用固定模板导出,按学生 / 课程或班级 / 日期 / 异常考勤四类模板输出;默认仅导业务结果层,服务财务统计与历史查询
- **协作与异常区**: 统一承接待老师确认、课后修改申请、暂无法确认异常、主管未响应、待更正与待冲正等任务流

**财务结算规则**:
- 运营结果实时更新,财务口径单独结算
- 每月 26 号进入待结算,需人工确认后生成当期财务结算结果
- 若 26 号仍有未收口异常,允许按已确认数据先结算,未确认部分自动进入"待更正/待冲正清单"
- 26 号后发现上月问题,统一走下月更正/冲正,不直接回写已结算数据

**待更正清单规则**:
- 一条原始事实记录一行,详情页下挂多轮更正时间线
- 默认只显示需要动作的记录,并按风险和时效优先排序
- 每条记录有明确负责人,系统自动分配,允许转交;转交需原因、接手确认、超时失效与失败留痕
- 财务只审核财务口径,不能直接修改业务事实;驳回后回流管理端补事实或补材料
- 提交财务审核前自动附标准化事实依据包;送审后冻结为审核快照,如有新事实需撤回并重提新版本

**主管兜底与预警**:
- 主管临时接手并闭环属于异常管理动作,需填写结构化原因并通知原负责人
- 主管兜底作为运营台与复盘的异常信号,第一版按负责人维度统计
- 近 7 天内同一负责人触发 3 次主管兜底时,向直属主管发预警
- 若直属主管持续未回执,进入"主管未响应"异常,运营侧可催办但不能代替主管回执;运营侧催办后仍无回执,再升级给直属主管的上级

#### 场景六:管理员补录请假(家长反馈)

**方式二:通过未到学生列表批量处理**

## 成功标准

1. **搜索准确性**:管理员输入拼音首字母或同音字,能准确匹配到目标学生/老师/课程
2. **考勤效率**:老师完成一个班级(25 人)的点名时间 < 3 分钟
3. **移动端导出可用性**:管理员导出的文本可直接粘贴到微信群,无需手工编辑
4. **Web 端结构化导出可用性**:财务和管理端可按固定模板导出结构化业务结果,无需手工整理
5. **时间权限准确性**:系统严格按照课程实例、课后申请、协作请求与财务结算规则控制权限,无越权操作
6. **财务结算清晰度**:运营结果与财务口径清楚分离,每月 26 号结算与下月更正/冲正链路可追溯
7. **空间利用率**:网格布局相比列表布局,屏幕可见学生数量提升 50% 以上

## 我注意到的设计要点

在这次需求对齐过程中,我注意到几个关键的设计决策:

1. **你对"空间效率"的重视** - 多次强调"一个学生占用一整行实在浪费空间",这说明你很清楚移动端屏幕的宝贵性,以及管理员/老师需要快速浏览大量学生信息的场景。网格布局是正确的方向。

2. **你对"实际工作流"的理解** - 你没有把"导出"粗暴理解成一个按钮,而是拆成了"移动端即时复制发群"和"Web 端结构化导出做财务统计"两类真实工作流,这让产品边界清楚很多。

3. **你对"容错性"的追求** - 同音字匹配("李嘉树"匹配"李佳澍")不是常见需求,但你主动提出,说明你真正理解了"信息碎片化"场景下的痛点:管理员面对口齿不清的低龄学生,需要系统有足够的容错能力。

4. **你对"权限边界"的清晰认知** - 你把老师端课中修改、课后申请解锁、管理端课中协作、财务只审口径不改事实这些边界逐层分开,这让复杂流程依然能落成清楚的责任链。
