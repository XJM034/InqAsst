# 接口与 Mock 冲突：`/admin/control/[classId]`

## 1. 冲突等级

`中`

这页的主要冲突不是对象完全错位，而是：

- 真页已经接上真实提交链路
- 页面状态比 mock 更重
- `未点名` 这个真实状态需要和管理端编辑规则一起重新解释

## 2. mock 依赖的对象

mock 班级详情页依赖的是“点名名单操作对象”。

它强调的是：

- 快速看名单
- 快速批量改状态
- 在一页内完成确认

## 3. 当前接口实际使用的对象

当前链路是：

- `fetchCourseDetail`
- `fetchAdminCourseStudents`
- `fetchAdminCourseLatestAttendance`
- `submitAdminAttendance`

这已经是一条真实可提交链路。

## 4. 本页当前应采用的状态规则

结合当前业务补充，管理端这里应采用：

- 管理端能展示所有状态：`已到 / 未到 / 请假 / 未点名`
- 管理端可编辑状态只有：`已到 / 未到 / 请假`
- `未点名` 可以展示，但不需要强行和 mock 完全一致
- `未点名` 不应作为“可切回去”的编辑选项

也就是说：

- `未点名` 是真实链路里的展示态
- 但不是管理端详情页里的目标编辑态

## 5. 具体冲突点

### 5.1 状态展示与状态编辑混在一起

真实数据里有：

- `unmarked`

如果直接把它既当展示态又当编辑态，页面就会出现：

- 可以把学生改回未点名

这和当前管理端规则不一致。

### 5.2 页面会比 mock 更重

当前页面除了名单和批量操作，还带着：

- 实时提交
- 提交失败提示
- 状态回滚

这些都是真实链路必须有的，但视觉上要尽量保持轻。

### 5.3 批量操作仍应是主流程

mock 更像“批量名单操作页”。

所以即使保留真实提交能力，也不应让页面变成“逐条编辑器”。

## 6. 处理建议

- 保留真实提交能力。
- 保留 `未点名` 的展示与统计。
- 管理端下拉和批量操作只允许改到：`已到 / 请假 / 未到`。
- 若学生当前状态是 `未点名`，可显示为 `未点名`，但不允许直接选回该状态。
- 空数据或加载失败时，保留正常页面骨架，不要退化成整页占位文案。
