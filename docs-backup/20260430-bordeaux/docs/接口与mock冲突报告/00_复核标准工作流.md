# 复核标准工作流

## 0. 配套关系
这份文档只负责“怎么判通过 / 不通过”，不负责“怎么改”。

配套关系固定为：

- `00_页面对齐工作流.md`
  - 负责“怎么实施”
- `00_推演事故防呆补充.md`
  - 负责“怎么防止推演先跑偏”

复核时不能只拿当前页面源码和测试就下结论，必须把这三份文档当成同一套约束来执行。

尤其是：

- 只要页面存在写动作，就必须同步复核“提交对象是否与页面业务对象一致”
- 这一条没过，即使链路通、测试过，也不能判通过

## 1. 目标
复核时只回答一个问题：

- 这个页面现在到底能不能算通过

这份工作流专门防下面几类误判：

- 只看页面文本，没看控件真实值
- 只看前端页面壳，没看真实后端链路
- 只看测试通过，没看页面实际行为
- 只看“链路存在”，误判成“功能可用”
- 只看“能提交”，没看“提交对象对不对”
- 只看交互页长得像，没看控件是否真的能交互

## 2. 复核结论格式
每页复核后只允许给下面三种结论之一：

- `通过`
- `不通过`
- `有条件通过`

`有条件通过` 只用于：

- 该页被明确约定为静态页
- 某个动作被明确约定暂不接真实提交，只校验页面壳

除此之外，不允许使用“基本通过”“大体没问题”“主体完成”这种模糊说法。

## 3. 复核前置输入
开始复核前，必须同时拿到：

1. 对应页面冲突报告
2. `teststaticPage` 对应页面
3. `http://localhost:3001` 对应页面
4. 当前前端页面源码
5. 当前前端读链路
6. 当前前端写链路
7. 当前后端真实读接口
8. 当前后端真实写接口
9. 对应测试文件

只要工作区同时存在前端和后端工程，就不能跳过 5-8。

## 4. 复核顺序
每个页面都按下面顺序复核，不能跳步。

### 4.1 先看基准页
先确认：

- 标题是否一致
- 主要卡片是否一致
- 列表组织方式是否一致
- 主要按钮文案是否一致
- 空态结构是否一致

只要和基准页明显不一致，就先记为疑点，不能先给通过。

### 4.2 再看真实页面行为
不能只读源码，至少要明确复核：

- 首次渲染是什么样
- 异步数据回来后页面会不会变化
- 出错时是否仍保持正常骨架
- 关键动作点击后是否真有后续行为
- 交互控件是否真的能点、能滑、能选、能改变状态

### 4.3 再看前后端链路
至少明确：

- 页面读的是哪个前端 loader / service
- 前端 loader / service 对应哪个后端接口
- 页面写动作调用哪个前端 submit
- 前端 submit 对应哪个后端写入口

如果某个动作没有真实后端入口，就不能把它复核成“可用写动作”。

### 4.4 再看提交对象是否正确
只要页面有写动作，必须单独检查：

1. 页面到底在改哪个业务对象
2. 页面改动的生效范围是什么
3. 页面上是否直接表达了这个对象或范围
4. 用户是否显式选择了这个对象或范围
5. 提交参数是否混入了只是技术锚点、但页面业务没表达的字段

出现下面任一情况，直接判 `不通过`：

- 页面表达的是规则，提交却落单条实体 ID
- 页面表达的是聚合范围，提交却落某个锚点记录
- 页面没有选择动作，却提交了对象主键
- 页面没有任何元素表达该对象，却把该对象作为主提交参数

### 4.5 最后看测试
测试只用于加固结论，不能代替页面复核本身。

要确认：

- 测试断言的是不是这页最关键的行为
- 测试有没有只断言标题和按钮，却没断言真实功能
- 测试是不是还锁着旧语义或旧文案

## 5. 页面类型复核标准

### 5.1 列表页
至少复核：

- 列表分组维度是否正确
- 主文案是否正确
- 副文案是否正确
- 搜索是否按正确索引工作
- 空列表时是否显示正常空态

### 5.2 详情页
至少复核：

- 页面标题
- 上下文卡片
- 当前状态展示
- 主要动作按钮
- 返回链路

### 5.3 编辑页
编辑页不能只看文字，必须额外复核控件默认值。

至少复核：

- 输入框是否默认回显真实值
- 下拉框是否默认选中真实值
- 单选 / 开关是否默认落在真实状态
- 异步数据回来后，控件值是否会同步更新
- 提交时是否真的把修改后的值发到后端

只要编辑页没有默认回显，一律直接判 `不通过`。

### 5.4 新增页
至少复核：

- 顶部上下文是否保留
- 空表单是否合理
- 必填校验是否存在
- 提交是否有真实写入口

### 5.5 选择页
至少复核：

- 默认选中态是否正确
- 搜索是否作用在正确对象上
- 分页 / 加载更多是否真实生效
- 提交选择后是否能落到真实后端
- picker / 滚轮 / 选择器是否真的可滑动或可点选，不是静态壳

### 5.6 静态页
只有明确约定为静态页，才允许不校验真实提交。

静态页至少复核：

- 页面壳
- 文案
- 上下文
- 返回链路
- 空态

## 6. 中文文案专项复核
复核中文时，不能只看“好像显示出来了”，要明确判断：

1. 页面实际渲染结果有没有乱码
2. 源码里有没有乱码字面量
3. 源码里有没有大量 `\uXXXX` 转义中文继续扩散
4. 测试断言是不是还在用旧乱码或旧转义

如果这一页本轮出现过编码污染，复核时必须同链路检查：

- 页面文件
- 共用组件
- `service / adapter`
- `tests/unit`
- `tests/service`

## 7. 不通过的高优先级判定
出现下面任一情况，直接判定 `不通过`：

1. 页面与基准页主要结构不一致
2. 页面运行时仍依赖 mock fallback
3. 页面关键写动作没有真实后端入口
4. 编辑页没有默认回显
5. 关键搜索索引不对
6. 页面存在明显乱码或编码污染
7. 页面失败后切成另一张完全不同的壳
8. 页面测试只覆盖标题，不覆盖关键功能
9. 提交对象与页面业务对象不一致
10. 交互页的关键控件实际上不能交互

## 8. 最小复核清单
每页复核时至少写下这 9 项：

1. 页面类型
2. 基准页是否一致
3. 中文文案是否正常
4. 读链路是否真实
5. 写链路是否真实
6. 提交对象是否正确
7. 关键控件或关键列表行为是否正常
8. 对应测试是否覆盖关键行为
9. 最终结论：`通过 / 不通过 / 有条件通过`

## 9. 一句话版本
复核不是确认“代码像不像改过了”，而是确认“页面是否和基准一致、关键行为是否真的可用、前后端链路是否真实、提交对象是否和页面业务一致、测试是否锁住了真正重要的点”。
