# 页面对比报告：`/admin/course-settings`

## 1. 先说结论

这一页是当前项目里偏移最严重的页面之一。

问题不是“UI 细节没抄到位”，而是页面业务含义已经被整体替换了。mock 分支里的这页本质上是“当天生效课程规则配置页”；当前接口版则被做成了“课程列表 + 老师设置入口页”。

这意味着如果不先把这一页的语义拉回去，后面学生名单、临时新增、保存动作、首页角标、教师端点名入口，都会继续偏。

## 2. mock 页面到底在表达什么

mock 里的 `/admin/course-settings` 想解决的是“今天到底按什么课程集合运行”。

它包含四个连续动作：

1. 先决定今天采用哪套规则行课。
2. 再看到今天最终生效的课程集合。
3. 再对这个集合做临时新增或移出。
4. 最后统一保存，作为老师端和管理端点名的共同基准。

所以这页并不是普通课程列表页，而是“当天课程规则总配置页”。

### 2.1 顶部区域表达的是“当天规则”

mock 顶部不是在问“点名方式是什么”，而是在问：

- 今天按默认星期规则行课，还是
- 今天整体按别的行课日来跑

这是一种“全局规则选择”。

### 2.2 中间列表表达的是“今日生效课程”

列表标题不是“全部课程”，而是“今日生效课程”。

这说明页面关心的不是系统里有哪些课，而是“今天真正参与运行的课程集合”。

### 2.3 卡片操作表达的是“把课留在今天，还是移出今天”

mock 卡片上的次按钮语义类似：

- 移出今日行课
- 保留到今日行课

这类动作都在强调“今天的最终课程集合”。

### 2.4 底部保存表达的是“总配置提交”

mock 底部那句总结文案非常关键，意思是：

- 今天最终生效多少门课
- 保存后老师端和点名总控都会用这一份结果

这说明整页是一个总提交页面，不是分散跳转页。

## 3. 当前接口版实际做成了什么

当前实现主要在：

- `app/admin/course-settings/page.tsx`
- `components/app/admin-course-settings-client.tsx`
- `lib/services/mobile-app.ts` 中的 `getAdminCourseSettingsData`
- `lib/services/mobile-adapters.ts` 中的 `mapAdminCourseSettingsData`

当前数据链路是：

- `getAdminCourseSettingsData`
- `fetchAdminTeacherSettingsOverview(getWeekStart())`
- `mapAdminCourseSettingsData`

也就是说，这页当前其实是拿“老师设置总览”在硬适配“课程设置页”。

于是页面自然被带成了另一套语义：

- 顶部变成“点名模式”
- 列表变成“课程列表”
- 卡片右上角显示老师信息
- 次按钮变成“老师设置”
- 顶部操作按钮跳去 `/admin/emergency`

## 4. 页面展示差异

### 4.1 页面定位完全变了

mock 要做的是：

- 选今天的规则
- 定今天的课程集合
- 一次性保存

当前页做成了：

- 看有哪些课
- 看每门课的老师
- 跳去做临时代课老师设置

这不是样式差异，是页面目标差异。

### 4.2 顶部规则区完全不对

mock 顶部应该是“当天规则”。

当前页顶部却是“点名模式”，而且只有“手动点名”这类概念。

问题在于：

- “手动点名”不等于“今天按哪套规则行课”
- 它无法回答今天课程集合是怎么生成的
- 它也无法对应 mock 的两种规则选项

### 4.3 搜索维度偏了

mock 搜索框表达的是：

- 搜课程名
- 搜上课地点

当前页更偏：

- 搜课程
- 搜老师

这说明页面视角从“课程安排”偏成了“老师设置”。

### 4.4 列表标题偏了

mock 标题是“今日生效课程”。

当前标题更接近“课程列表”。

一个是“今天的最终结果”，一个是“系统里有哪些项目”，语义完全不同。

### 4.5 卡片摘要内容偏了

mock 卡片摘要重点应该是：

- 上课时间
- 上课地点
- 老师
- 人数
- 当前为什么出现在今天

当前卡片更像：

- 课程名
- 老师名
- 某些老师设置状态

其中最关键的问题是，当前卡片缺少“为什么这门课属于今天”的解释。

### 4.6 次按钮语义彻底变化

mock 次按钮要表达的是：

- 是否保留到今日行课
- 是否移出今日行课

当前按钮却变成：

- 老师设置

这直接把页面从“课程集合配置”切换成“老师配置入口”。

### 4.7 顶部主操作也偏了

mock 顶部主操作是“临时新增课程”，它在改今天的课程集合。

当前主操作跳去 `/admin/emergency`，更像“去设置老师”。

### 4.8 底部保存虽然还在，但已经失去真实含义

当前界面可能保留了“保存课程设置”按钮，但前端数据模型里并没有完整承接：

- 当天规则
- 今日生效课程集合
- 临时新增集合
- 临时移出集合

所以这个保存动作已经没有 mock 里的总提交语义。

## 5. mock 与实际接口数据差异

### 5.1 mock 数据模型

mock 使用 `adminCourseSettingsData`。

它天然包含或隐含的是：

- 当天规则选项
- 今天最终生效课程列表
- 课程状态标签，如今日行课、临时新增
- 支持新增、移出、保留的动作语义
- 底部总提交信息

### 5.2 当前接口数据模型

当前页使用：

- `getAdminCourseSettingsData`
- `fetchAdminTeacherSettingsOverview`
- `mapAdminCourseSettingsData`

老师设置总览接口擅长提供的是：

- 某周内课程与老师的对应关系
- 默认老师或临时代课老师
- 课节层面的老师配置状态

它并不天然提供 mock 所需的核心字段。

### 5.3 当前接口缺的不是小字段，而是整套业务对象

当前适配里最缺的是：

- 今天整体采用哪套规则
- 今日最终生效课程集合
- 每门课是默认生效还是临时新增
- 每门课是否被临时移出
- 保存后要统一影响哪些下游页面

所以问题不是 adapter 没写全，而是“拿错了上游对象”。

### 5.4 这页为什么会越接接口越不像

因为一开始 mock 页面表达的是“课程规则”。

但真实接入时，前端拿到的最容易用的数据却是“老师设置总览”。

一旦用这个总览去喂页面，页面自然会朝下面方向变形：

- 关心老师，而不是关心今天规则
- 关心课节老师配置，而不是关心今天课程集合
- 关心跳转配置页，而不是关心整页保存

## 6. 对齐建议

### 6.1 先恢复页面定位

先明确这页的唯一中心语义：

- 配置今天最终生效的课程集合

不是：

- 展示全部课程
- 展示老师配置入口

### 6.2 重建页面数据模型

至少要补出下面这些前端对象：

- `dayRule`
- `effectiveCourses`
- `temporaryAddedCourses`
- `temporarilyRemovedCourseIds`
- `summaryBeforeSave`

如果后端暂时没有整包接口，也要在前端聚合，不要再拿 `TeacherSettingsOverview` 生拼。

### 6.3 把老师设置从主语义降级成附属信息

老师当然可以显示，但只能作为课程摘要的一部分，不能把整页变成“老师设置入口页”。

### 6.4 恢复总保存语义

页面必须能真正表达：

- 本次改动前后的课程集合变化
- 保存后会影响老师端点名和管理端总控

否则“保存课程设置”只是一个外观残留按钮。

## 7. 对齐时要重点核对的点

- 顶部是不是“当天规则”，而不是“点名模式”。
- 列表是不是“今日生效课程”，而不是“课程列表”。
- 顶部主按钮是不是“临时新增课程”，而不是跳去老师设置链路。
- 课程卡片次按钮是不是“移出今日行课/保留到今日行课”，而不是“老师设置”。
- 底部保存是不是总提交动作，而不是空壳按钮。
