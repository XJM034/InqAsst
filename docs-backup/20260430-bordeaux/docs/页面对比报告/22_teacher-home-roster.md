# 页面对比报告：`/teacher/home/roster`

## 1. 先说结论

这页的外观相对稳定，但它的依赖对象已经从 mock 的“课程演示上下文”，切换成了真实 `courseId/courseSessionId`。所以它的风险主要在链路语义，而不是视觉框架。

## 2. mock 页面想表达什么

mock 里的 roster 页更像：

- 当前还没到点名时间
- 先给老师看这门课的学生名单

它是教师首页进入的一张静态结果页，重点是“看名单”。

## 3. 当前接口版实际做成了什么

当前实现主要在：

- `app/teacher/home/roster/page.tsx`
- `lib/services/mobile-app.ts` 中的 `getAttendanceSession`

当前页面已经完全依赖：

- `courseId`
- `courseSessionId`

去读取真实名单数据。

## 4. 页面展示差异

### 4.1 表面结构仍接近 mock

- 课程摘要仍在
- 学生名单仍在
- 点名前查看名单的意图仍在

### 4.2 真实参数主导了页面存在方式

mock 里这页更多是某种稳定场景的延展页。

当前页已经必须绑在真实课节上下文上。

### 4.3 与教师首页的关系更紧

由于页面依赖真实 session，因此首页如果主课/代课判断有误，这里就会一起出错。

## 5. mock 与实际接口数据差异

### 5.1 mock 数据语义

mock 可以通过 `teacherHomeData + attendanceSession` 一类固定对象组合出稳定名单页。

### 5.2 当前接口数据语义

当前直接依赖 `getAttendanceSession(courseId, courseSessionId)`。

也就是说它只认真实课节，不认 mock 的演示上下文。

## 6. 对齐建议

- 保持当前外观结构。
- 重点校验它与教师首页、点名入口之间的 session 语义是否一致。
- 名单页文案仍要保留 mock 的“当前查看名单”而非“进入正式点名”氛围。

## 7. 对齐时要重点核对的点

- 从教师首页进入名单页时，课程上下文是否准确。
- 名单页是否仍然像点名前的查看页，而不是半个点名执行页。
- 返回路径是否自然回到教师首页或点名主链路。
