# 页面对比报告：`/teacher/attendance/no-class`

## 1. 先说结论

这页是当前接口版新增出来的正式页面，mock 中没有严格一一对应的独立点名无课页。因此它的任务不是“还原 mock 某张图”，而是确保新增页面不破坏 mock 教师端整体体验。

## 2. mock 页面想表达什么

mock 教师端确实有“无课”状态，但更多是通过现有页面文案和空态承接，不一定单独拆出一张点名无课页。

因此 mock 给这类场景的要求通常是：

- 简洁
- 不慌张
- 给老师清楚的下一步

## 3. 当前接口版实际做成了什么

当前实现主要在：

- `app/teacher/attendance/no-class/page.tsx`

它承担的是点名入口解析失败或今日无课时的落地页。

## 4. 页面展示差异

### 4.1 当前页属于新增承接页

它不是 mock 中的核心视觉页面，而是当前真实路由体系补出来的空态页。

### 4.2 关键看是否破坏教师端气质

如果无课页做得太像错误页，就会和 mock 的轻量教师端风格冲突。

### 4.3 页面文案要与首页、no-class 页统一

老师不需要知道“为什么 session 解析失败”，只需要知道：

- 今天没有可点名课程
- 下一步可以回首页或查看别的入口

## 5. mock 与实际接口数据差异

这页没有严格对等的 mock 数据模型。

它更多是当前真实路由体系的补充页面，因此数据差异不在字段，而在于：

- 是否和教师首页语义一致
- 是否与 `/teacher/no-class` 保持统一体验

## 6. 对齐建议

- 文案尽量贴近 mock 的空态语气，不要做成报错页面。
- 与 `/teacher/no-class` 做统一设计，不要出现两套无课风格。
- 给出明确但轻量的返回路径。

## 7. 对齐时要重点核对的点

- 页面是否轻量、稳定。
- 老师看到后是否知道今天就是无课，而不是系统异常。
- 与 `/teacher/no-class` 是否重复或冲突。
