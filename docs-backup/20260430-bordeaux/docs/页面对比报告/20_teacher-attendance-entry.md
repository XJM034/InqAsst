# 页面对比报告：`/teacher/attendance`

## 1. 先说结论

这页在 mock 分支里并不是一个正式视觉页，而当前接口版新增了一个真实中转入口页。它本身不是视觉偏移问题，而是教师点名链路被重构后的入口适配页。

## 2. mock 页面想表达什么

mock 里教师底部 tab 更像直接指向点名页本身，通常直接进入固定 demo 页面或稳定的点名目标页。

也就是说：

- 用户点击“点名”
- 直接看到点名页面

中间没有额外入口层。

## 3. 当前接口版实际做成了什么

当前实现主要在：

- `app/teacher/attendance/page.tsx`

它的角色是：

- 先解析今天是否有课
- 如果有课，重定向到具体 `courseId/courseSessionId`
- 如果无课，跳到 `/teacher/attendance/no-class`

## 4. 页面展示差异

### 4.1 这不是一张正式展示页

当前页更多承担路由分发功能，而不是视觉展示功能。

### 4.2 教师端入口语义被改成“自动解析”

mock 的入口更简单直接。

当前入口更智能，但也更依赖首页和 session 数据是否准确。

### 4.3 一旦首页语义错，这里也会跟着错

如果教师首页已经把第二条 session 当成代课，这个入口页就可能把老师带去错误的点名上下文。

## 5. mock 与实际接口数据差异

### 5.1 mock 数据语义

mock 并不需要这层中转数据模型。

### 5.2 当前接口数据语义

当前页实质上依赖：

- `getTeacherHomeData`
- 当日 session 判断

也就是说它本身没有独立页面数据，而是吃教师首页链路结果。

## 6. 对齐建议

- 保留中转能力，但不要让它改变 mock 的入口直觉。
- 首页语义修正后，这里也要同步校验重定向目标是否正确。
- 无课时跳转要和 mock 的无课页风格及文案保持一致。

## 7. 对齐时要重点核对的点

- 点击底部“点名”后是否总能进入正确课程。
- 无课时是否稳定跳去正确无课页。
- 中转过程是否对老师透明，不造成额外理解成本。
