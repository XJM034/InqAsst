## Problem Statement

本次设计不再延续上一版“查询 + 考勤”的双核心结构，而是收缩为更贴近业务交付的单一主线：

- 管理端先完成规则设置与当天调整。
- 老师端按最终生效规则完成点名。
- 管理端继续在移动端完成现场点名总控与复制同步。

本次设计以 [Attendance.pen](/Users/minxian/Documents/alex_project/InqAsst/Attendance.pen) 为原型基座继续演进，查询相关功能本阶段不开发，管理端首页定位改为“管理功能”。文档内统一优先使用“点名”表达业务含义，原型中历史遗留的“考勤”命名视为后续待替换文案。

本轮真正要解决的核心问题，不只是“纸质流程麻烦”，而是管理端无法在上课开始后的前几分钟及时知道哪些学生未到。当前流程下，老师先在纸质点名册勾选，助教再誊写到在线表，管理端往往要等到开课 20 分钟左右才第一次看到未到学生；再去联系班主任确认去向，收到反馈时可能已经过去 40 分钟。如果学生是逃课、走失或发生其他意外，这个延迟窗口会直接削弱现场处置能力。

## Demand Evidence

如果本次能力不能上线，现场仍需沿用纸质点名册 + 在线文档 + 群同步的串联方式：

- 助教老师先把主纸质点名册交给授课老师。
- 授课老师根据实际上课情况在纸质点名册上勾选学生。
- 助教老师再把纸质结果誊写到在线文档。
- 管理员老师查看在线文档后，再向班主任同步确认。
- 等班主任回复后，管理员才知道学生是请假、迟到还是去向未明。

这说明当前最真实的痛点不是“没有记录工具”，而是：

- 管理端无法在开课后的前几分钟及时看到未到学生，缺少第一时间介入窗口。
- 纸质勾选 -> 助教誊写 -> 在线文档查看 -> 班主任确认，这条链路过长，导致真实未到信息通常滞后 20 到 40 分钟。
- 一旦学生属于逃课、走失或其他异常情况，这种滞后并不只是效率问题，而是现场风险控制问题。
- 规则分散，老师现场拿到的信息可能已经过时。
- 管理员现场需要快速调整班级、老师、时间，但现在没有一个统一、可登录、可即时生效的设置入口。

## Status Quo

现阶段项目已具备老师端与管理端移动端原型，其中管理端移动端已经形成较明确的现场操作链路：

- 班级列表
- 未到学生列表
- 复制未完成班级
- 复制未到学生
- 收到班主任回复后直接修改未到状态

这些能力在 [Attendance.pen](/Users/minxian/Documents/alex_project/InqAsst/Attendance.pen) 中已有较成熟的表达，因此本轮不应推翻移动端点名总控逻辑，而应把新增重点放在“管理设置能力”上，尤其是 Web 端的结构化设置中心，以及移动端的应急补充设置。这样做的目标不是单纯让操作更顺，而是尽可能把“发现未到学生”的时间点从开课后 20 分钟以后，前移到老师提交点名后的第一时间。

## Target User & Narrowest Wedge

### Target User

- 校区管理老师：负责维护和更新总规则、调整当天安排、处理代课、现场督促老师完成点名，并把未到学生信息同步给相关群。总规则不要求在每次上课前重新导入，管理老师的主要工作是基于现有规则做更新和调整。
- 外部上课老师：通过手机号登录 Attendance app，查看自己当日需要负责的班级与学生，并在规定时间内完成点名。

### Narrowest Wedge

第一版最小可交付闭环定义为：

- 管理员可基于已经维护在系统中的课程总规则，完成当天生效规则的设置和调整；总规则由学生数据库周期性更新，不要求校区老师在每次上课前重新导入。
- 老师可基于最终生效规则，在手机端看到自己的上课时间、地点、班级与学生名单，并在规定时间内完成点名提交。
- 管理员可继续在移动端汇总点名结果，一键复制未完成班级文本、未到学生文本到助教群或授课老师群，并在收到班主任回复后直接修改学生未到状态。
- 线下群同步和纸质点名册修订仍可继续保留，不强行要求系统内完成所有后续状态闭环。

该闭环的业务价值在于，让管理端尽早知道“哪些学生未到”，从而把联系班主任、核实去向的动作尽量前移，而不是继续等纸质信息层层传递。

## Constraints

- 本期只考虑管理端和老师端，不恢复查询功能。
- 管理端既要适配 Web，也要保留移动端；但两端职责不同，不能简单要求“移动端完整复制 Web 端全部设置能力”。
- 老师端本期仍以移动端为主，不要求适配 Web。
- 管理设置必须围绕真实业务闭环，不做过多功能延展，不扩展成复杂教务系统。
- 文档只描述需求、页面结构、信息层级和交互边界，不讨论代码实现。

## Premises

1. 本期范围收缩为“管理端设置 + 老师端点名”，不开发独立查询功能，文案统一优先改为“点名”。
2. 管理端的基础总规则不要求手工从零录入，也不要求校区老师在每次上课前重新导入；它以现有学生数据库同步的课程安排为基础长期维护，管理员主要是在此基础上做更新和当天调整。总规则内容至少包括周几几点、上课地点、课程老师、课程内学生 ID、课程内学生姓名、学生所属行政班。其中上课地点、学生 ID、学生姓名、行政班来自系统同步；本期不在系统内维护班主任字段，后续沟通以行政班为准。
3. 管理端移动端继续承担现场点名汇总与复制动作；新增 Web 端的核心目标是更高效地完成“管理设置功能”，同时移动端保留轻量应急设置能力。

## Approaches Considered

### Approach A: 最小补丁版

**Summary**: 在现有移动端原型上小幅补充设置入口，不单独建立清晰的 Web 设置中心。

**Effort**: S

**Risk**: Low

**Pros**:

- 改动最小，能快速维持现有原型结构。
- 开发风险最低，适合极短周期交付。
- 管理端移动端操作成本低，不需要重讲太多新结构。

**Cons**:

- “总规则 + 当天调整 + 应急修改”的关系会模糊。
- Web 端存在感弱，无法真正体现“管理设置效率提升”。
- 后续需求继续增加时，很容易再次重构。

### Approach B: 设置中心版

**Summary**: 以 Web 端作为主要管理设置中心，清晰拆分上课时间、课程、老师三类设置；移动端保留现场点名总控，并补足单条应急修改能力。

**Effort**: M

**Risk**: Low-Med

**Pros**:

- 最符合本轮业务变化，Web 端价值清晰。
- 总规则、统一切换、临时新增、老师调整之间的逻辑更容易讲清楚。
- 移动端和 Web 端职责清楚，不会互相打架。

**Cons**:

- 需要补一套完整的管理设置结构。
- 页面和状态说明比最小补丁版更多。
- 需要明确哪些设置属于 Web 主操作，哪些设置只在移动端做应急处理。

### Approach C: 双端运营台版

**Summary**: Web 端同时承担设置与当日运营总览，移动端也扩展更多设置能力，双端都尽量完整。

**Effort**: L

**Risk**: Med

**Pros**:

- 表面上功能最完整。
- 后续看似更容易承接更多管理需求。
- 电脑端和手机端都能承担更多角色。

**Cons**:

- 容易偏离“不要过多功能延展”的要求。
- 双端边界会变重，反而降低产品清晰度。
- 与当前交付目标相比明显过度设计。

## Recommended Approach

推荐采用 **Approach B: 设置中心版**，并补充一条关键原则：

- Web 端是管理设置的主操作端。
- 移动端不是设置能力空白，而是保留“单条、现场、应急”的轻量设置能力。

这意味着：

- 批量、结构化、计划性的设置动作优先在 Web 端完成。
- 当天现场突发问题，如代课老师手机号录错、临时需要替换老师、需要对单个班级做快速修正时，管理员可直接在移动端应急修改，不必等到电脑端处理。

## Detailed Requirements

### 一、全局范围与命名

- 本次设计基于 [Attendance.pen](/Users/minxian/Documents/alex_project/InqAsst/Attendance.pen) 继续修改。
- 查询相关页面和流程不再继续扩展。
- 管理端首页目标由“查询入口”调整为“管理功能入口”。
- 文档中统一使用“点名”称呼；原型图中已有的“考勤”文案可视为待统一替换项。
- 本期角色保持双角色：老师端、管理端。

### 二、规则来源与生效逻辑

#### 2.1 总规则来源

- 总规则以现有学生数据库同步的课程数据为基础。
- 数据获取方式为：从包含学生数据的数据库中，按校区定时抓取该校区全部课程的基础数据。
- 总规则是长期维护的默认课程底盘，不要求校区老师在每次上课前重新导入一次。
- 校区管理老师的主要动作是更新总规则和处理当天调整，而不是反复重建全量规则。
- 总规则至少包含以下信息：
- 课程周几上课。
- 课程具体上课时间与下课时间。
- 课程上课地点。
- 课程对应老师。
- 课程内学生 ID。
- 课程内学生姓名。
- 学生所属行政班。
- 其中上课地点、学生 ID、学生姓名、行政班来自系统同步。

#### 2.2 规则层级

本期需要明确三层规则优先级：

1. **总规则**：由数据库同步维护的默认课程安排。
2. **当天规则调整**：管理员基于总规则对某一天的整体安排做变更。
3. **现场应急调整**：管理员针对单个班级或单个老师做即时修正。

生效顺序为：现场应急调整优先于当天规则调整，当天规则调整优先于总规则。

#### 2.3 当天规则调整的核心场景

管理员至少可完成以下几类调整：

- 调整上课时间，如整体提前或延后。
- 支持课程换日，例如原本周三上课的课程，本周需要改到周二上；调整后老师端应同步显示该课程已转到周二，并显示周二对应的学生信息。
- 未做课程调整时，老师端默认按当天星期显示对应点名信息；例如当天为周二，则默认显示周二课程。
- 一旦管理端做了课程调整，则管理端和老师端都必须以调整后的规则为准；例如周二调整为按周三课程行课，则两端都显示周三课程，并明确提示“课程调整，本次按照周三课程行课”。
- 当天沿用默认星期规则，例如周二默认使用周二班级行课。
- 将某一天的班级统一切换为另一星期的班级安排，例如周二统一切换为周三班级上课。
- 在默认或切换后的基础上，自定义新增总规则中原本不存在的行课课程。
- 支持在特定时间新增最多 10 个课程的临时行课安排。

### 三、管理端产品定位

#### 3.1 管理端 Web

定位为 **管理设置主操作端**，适合处理：

- 查看当前总规则及其最近更新结果。
- 对当天行课规则做结构化调整。
- 批量查看和编辑课程、时间、老师关系。
- 清晰核对某一天最终生效的课程与老师安排。
- 在老师设置中按班级列表补充当天实际负责老师，而不是从老师列表反向分配班级。

#### 3.2 管理端移动端

定位为 **现场点名总控端 + 应急设置端**，继续承担：

- 班级点名完成度查看。
- 未到学生查看。
- 一键复制未完成班级文本。
- 一键复制未到学生文本。
- 收到班主任回复后直接修改未到状态。

同时新增轻量应急设置能力，用于处理现场突发错误，不要求完整覆盖 Web 端全部设置能力。

### 四、管理端首页

#### 4.1 Web 端首页

管理端 Web 端定位改为”管理功能”，不再强调搜索。

Web 端采用左侧导航 + 右侧工作区的布局结构，左侧导航包含三类核心入口：

- 上课时间设置
- 课程设置
- 老师设置

左侧导航底部展示”今日摘要”信息，包括：

- 今日行课日期（如：2026-03-25 周二）
- 今日生效班级数量（如：8 个班级）
- 今日涉及老师数量（如：6 位老师）

右侧工作区根据左侧导航选择展示对应的设置页面内容。

#### 4.2 移动端首页

管理端移动端首页可轻量化，但也应能进入“管理功能”。

移动端首页不要求承担完整配置流程，但至少需要：

- 进入点名总控
- 查看今日生效规则摘要
- 进入应急设置入口

### 五、上课时间设置

#### 5.1 设置目标

管理员需要先定义“什么时候上课”，再让老师在对应时间完成点名。

#### 5.2 必要能力

- 查看默认时段规则。
- 调整某一天实际生效的上课时间。
- 配置老师点名时间要求和点名时间窗口。
- 明确老师需在上课开始后的规定时间内完成点名提交。
- 当实际上课时间被改动时，点名时间窗口也要同步改动，而不是继续沿用原时间。

#### 5.3 点名时间规则

- 管理端需要能配置每节课的点名时间窗口。
- 当上课时间调整时，点名时间窗口应联动更新。
- 例如：原定 16:00 上课，如当天改为 15:50 上课，则点名时间窗口应支持同步改为 15:40 - 16:00。
- 老师应在配置好的点名时间窗口内完成点名，业务上仍以尽快完成首轮点名为目标。
- 若老师超时未提交，系统应在老师端和管理端显示明显提醒。
- 超时提醒不必强制阻断提交，但必须形成高可见性提示，便于管理员督促。

#### 5.4 Web 端与移动端差异

- **Web 端**：在"上课时间设置"页面中统一展示周几选择、实际上课时间设置、点名时间限制规则，所有时间相关设置在一个页面内完成。
- **移动端**：将"设置实际上课时间"和"设置点名时间"拆分为两个独立页面，每个页面专注于单一设置任务，便于现场快速操作。

### 六、课程设置

#### 6.1 设置目标

管理员要能快速决定”今天到底哪些课程上课”。

#### 6.2 必要能力

- 查看某一天按总规则应当行课的默认课程集合。
- 一键沿用默认星期规则。
- 一键切换为另一星期的课程集合，例如周二切换为周三课程。
- 在已选课程集合的基础上继续人工增减课程。
- 在特定时间新增最多 10 个临时课程行课。

#### 6.3 课程信息要求

每个课程设置项至少应展示：

- 课程名称
- 上课时间
- 上课地点
- 当前负责老师
- 课程内学生人数
- 该课程所含学生的行政班信息来源

- 其中上课地点来自系统同步。

进入课程对应的学生名单详情后，字段至少应包括：

- 学生 ID
- 学生姓名
- 行政班
- 学生来源
- 老师端显示

- 其中学生 ID、学生姓名、行政班来自系统同步。

#### 6.4 今日生效规则视图

无论在 Web 端还是移动端，都需要能看见”今天最终生效的课程规则”，避免管理员设置后仍需要靠记忆判断。

- 默认情况下，今日生效规则应按当天星期直接展示。
- 如当天存在课程换日或统一切换，页面应明确标识调整结果，避免老师误以为仍按原星期上课。
- Web 端在”课程设置”页面中通过表格形式展示课程列表和相关信息。
- 移动端在”课程与班级设置”页面中通过卡片形式展示，包含”当天规则”模式选择和课程列表。

### 七、老师设置

#### 7.1 设置目标

管理员要能决定“今天每个班级由谁上课、谁可以登录、谁要去点名”。

#### 7.2 必要能力

- 查看每个班级的默认负责老师。
- 查看每个班级今天是否已补充负责老师。
- 为某个班级补充今天实际负责点名的老师。
- 优先从已有老师列表中选择负责老师。
- 当已有列表中没有该老师时，可临时新增代课老师。
- 为代课老师填写或修改手机号，以便登录 Attendance app。

#### 7.3 Web 端老师设置交互逻辑

`Web-老师设置` 应按班级列表组织，而不是按老师列表组织。

核心原因是：管理员在真实操作中通常是先找到“哪个班今天缺负责老师”，再补充该班今天的实际负责老师，而不是先找到某位老师再去分配班级。

因此页面结构应满足：

- 先展示今日上课班级列表。
- 每个班级行至少展示：
  - 班级名称
  - 上课时间
  - 上课地点
  - 默认负责老师
  - 今日补充老师状态
  - `补充老师` 操作入口
- `补充老师` 的主路径应是先从已有老师列表中选择。
- 当已有老师列表中没有该人时，再提供“临时添加代课老师”作为兜底路径。

页面文案必须明确：

- 默认负责老师由系统同步维护。
- 本页补充的是“今天实际负责点名的老师”。
- 该补充仅对今天生效，不改长期默认老师。

#### 7.4 补充老师与代课规则

- 如果今天补充的负责老师原本已在上课老师列表中，登录后应同时看到自己的原定上课信息和被分配的代课上课信息。
- 如果今天补充的负责老师不在现有老师列表中，管理员可临时新增老师姓名与手机号，使其能直接登录 Attendance app 完成点名。
- 补充负责老师默认是“按班级对今天生效”的临时安排，不应直接改写该班的长期默认负责老师。
- 老师手机号属于关键登录信息，必须支持修改。

#### 7.5 移动端应急改手机号

这是本期必须明确落地的移动端应急场景：

- 某位代课老师到校后，管理员发现此前录入手机号错误。
- 管理员无需回到电脑端，可在移动端直接修改该老师手机号。
- 修改后该老师即可使用正确手机号登录 Attendance app。

### 八、管理端移动端应急设置边界

移动端设置不要求完整复制 Web，但至少应支持以下“单条应急处理”：

- 修改单个班级的当前上课时间。
- 为单个班级新增代课老师。
- 修改代课老师手机号。

本期不要求移动端承担以下重型操作：

- 批量同步或更新总规则
- 大规模批量调整
- 复杂历史版本对比
- 长周期排课维护

这些仍以 Web 端为主。

### 九、管理端点名总控

本模块继续保留在移动端，原则上沿用 [Attendance.pen](/Users/minxian/Documents/alex_project/InqAsst/Attendance.pen) 中已有结构，不再另起一套复杂流程。

#### 9.1 班级列表

#### 9.2 未到学生列表

### 十、老师端点名设计

#### 10.1 端能力边界

- 老师端本期仍为移动端。
- 老师通过手机号登录。
- 老师只需要完成“看到自己该上哪节课、点哪个班、在规定时间内提交点名”这一核心任务。

#### 10.2 老师端首页信息

老师端首页采用综合信息展示方式，包含：

- **周选择器和周日历**：可查看和切换不同周的课程安排
- **今日行程区域**：展示当天的课程任务
  - 任务卡片：显示课程名称、上课时间、上课地点、学生人数等信息
  - 代课卡片：如有代课任务，用橙色左边框突出显示
- **明日预告**：提前展示明天的课程安排

老师应根据最终生效规则看到：

- 上课时间
- 上课地点
- 自己负责的班级
- 班级内学生名单

- 未做课程调整时，老师端默认按当天星期显示对应课程和点名信息。
- 如管理端对当天课程做了调整，则老师端不再显示原星期课程，而是显示调整后的课程内容。
- 当调整场景为”周二按照周三课程行课”时，老师端应明确提示：`课程调整，本次按照周三课程行课`。

如果老师既有自己的原定课程，又承担代课课程，应同时展示两类任务。

#### 10.3 点名页

老师端点名页包含两种状态：

**状态一：直接点名首页**
- 展示课程卡片和筛选选项
- 显示学生列表
- 可快速进入点名操作

**状态二：首次提交页面**
- 显示点名时间窗口条（如：15:50 - 16:10）
- 提供快速筛选功能
- 显示蓝色交互提示条：说明"点击学生卡切换已到 / 未到；黄色框为管理端修改，请老师注意是否属实"
- 学生列表采用网格布局展示
- 支持通过点击学生卡片快速标记已到 / 未到
- 管理端修改的学生用黄色框标识，提醒老师确认准确性
- 支持一键全选已到
- 展示实时统计
- 底部提供提交按钮
- 提交前进行状态确认（通过弹窗确认）

#### 10.4 提交后边界

- 老师完成点名提交后，视为完成系统内核心动作。
- 管理员后续在移动端继续同步未到学生，并在收到班主任回复后直接修改未到状态，属于管理端后续动作。
- 不强行要求老师根据群消息再次回到系统内更新最终结果。
- 纸质点名册的最终修订仍可在线下完成。

### 十一、学生管理功能

原型中包含完整的学生管理功能，用于维护课程内的学生名单。

#### 11.1 功能范围

**Web 端：**
- 学生名单详情页：展示课程摘要、学生列表、支持搜索和筛选
- 学生信息字段：学生 ID、学生姓名、行政班、学生来源、老师端显示状态

**移动端：**
- 学生名单查看
- 新增学生：单个添加学生到课程
- 批量导入学生：支持批量导入学生信息
- 编辑学生：修改学生信息

#### 11.2 导入字段说明

批量导入支持的字段包括：
- 学生 ID
- 学生姓名
- 行政班
- 班主任（可选，当前列表默认不展示）
- 学生来源

其中学生 ID、学生姓名、行政班来自系统同步，也可通过导入或手工录入补充。
