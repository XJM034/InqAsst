# 课程设置空白课程创建协同

日期：2026-04-21

适用对象：`daoleme` 后端工程师

## 1. 当前问题

管理端页面：

- `/admin/course-settings/new-course`

当前页面已经支持两种入口：

- 从已有课程添加
- 添加空白课程

其中“从已有课程添加”可以正常继续沿用已有链路；
但“添加空白课程”当前在共享 dev 口径下提交后，页面只收到一条后端报错：

- `不能为null`

这条报错对前端和测试都不够用，因为它没有指出到底是哪个字段被判成了 `null`。

## 2. 页面实际复现

2026-04-21 当前前端页面已切到新的业务口径：

- 课程名称仍由管理员填写
- 开始时间 / 结束时间不再手填
  - 直接读取当天“时间设置”里的实际上课时间
- 点名时间只读展示
  - 已配置：直接显示时间区间
  - 未配置：显示 `未配置`
- 学生必须至少添加 1 名
  - 系统内学生：搜索当前校区已有学生
  - 系统外学生：填写姓名并指定行政班
- 老师支持两种来源
  - 系统内老师：从当前校区老师列表里选择
  - 系统外老师：手填姓名 / 手机号

当前共享 dev 口径下，只要走“空白课程”提交，后端仍可能返回：

- `不能为null`

## 3. 前端当前发送的请求体

当前前端代码位置：

- `app/admin/course-settings/new-course/page.tsx`
- `lib/services/mobile-client.ts`

当前“空白课程”提交会调用：

- `POST /api/admin/course-settings/temporary-courses`

请求体当前为：

```json
{
  "date": "2026-04-21",
  "courseName": "周二篮球补课_test",
  "sessionStartAt": "2026-04-21T18:00:00+08:00",
  "sessionEndAt": "2026-04-21T19:30:00+08:00",
  "location": "篮球场",
  "teacher": {
    "source": "INTERNAL",
    "teacherId": 88
  },
  "students": [
    {
      "source": "INTERNAL",
      "studentId": 7
    },
    {
      "source": "EXTERNAL",
      "name": "赵小明",
      "homeroomClassId": 10
    }
  ]
}
```

补充说明：

- `sessionStartAt / sessionEndAt` 不是页面手填，而是前端根据 `GET /api/admin/time-settings/sessions?date=...` 读取当天配置后自动带出
- 点名时间当前只做展示，不随本次创建请求一起提交

## 3.1 当前新表单额外依赖的读链路

除了最终创建接口之外，当前“空白课程”页还依赖以下后端读接口：

- 系统内老师选择：
  - `GET /api/admin/teachers`
  - 用途：加载当前管理员所在校区的系统内老师列表
- 系统内学生选择：
  - `GET /api/admin/me`
  - 用途：读取当前管理员的 `campusId`
  - `GET /api/admin/campuses/{campusId}/students/search?q=...&limit=10`
  - 用途：按当前校区实时搜索系统内学生

如果上述任一接口在共享 dev / 本机联调口径下：

- 返回空数据但实际校区有老师 / 学生
- 返回的不是当前校区数据
- 缺字段，例如 `campusId / teacherId / studentId / name`
- 直接报错或超时

那么当前前端页面上的这些能力就都会失效：

- 不能正常选择系统内老师
- 不能正常搜索系统内学生
- 即使最终创建接口已支持，新表单也无法完整走通

## 4. 当前判断

当前更像是“前后端契约未对齐”，不是用户漏填。

原因：

- 前端本地校验已经保证：
  - 课程名不为空
  - 实际上课时间已从时间设置读取
  - 学生至少 1 人
  - 系统外学生必须带行政班
- 但后端返回的仍是泛化的 `不能为null`
- 当前已知老链路只支持：
  - `POST /api/admin/course-settings/temporary-courses`
  - body：`{ date?, sessionId }`

因此高概率是下面两种情况之一：

1. 后端当前仍只支持老的 `sessionId` 入参，不支持“空白课程”创建
2. 后端已支持“空白课程”创建，但字段名 / 嵌套结构不是前端当前发送的这组

## 5. 需要后端明确的内容

请后端明确以下任一方案：

### 方案 A：复用现有接口

继续使用：

- `POST /api/admin/course-settings/temporary-courses`

但明确“空白课程”模式的正式请求体，至少说明：

- 哪些字段必填
- `sessionStartAt / sessionEndAt` 是否继续由前端透传配置后的实际上课时间
- `teacher` 是否接受以下两种结构：
  - 系统内：`{ "source": "INTERNAL", "teacherId": number }`
  - 系统外：`{ "source": "EXTERNAL", "name": string, "phone"?: string }`
- `students` 是否接受以下两种结构：
  - 系统内：`{ "source": "INTERNAL", "studentId": number }`
  - 系统外：`{ "source": "EXTERNAL", "name": string, "homeroomClassId": number }`
- `date` 是否必填
- 点名时间未配置时是否允许创建
- 创建成功后返回是否仍为完整 `overview`

### 方案 B：单独新增接口

如果后端不希望把“空白课程”塞进 `temporary-courses`，请直接给出新的专用接口：

- 路径
- 请求体
- 返回体
- 错误码 / message

## 6. 前端需要的最小保证

为了让前端可以尽快对齐，至少需要后端补齐这些信息：

- 一个确定可用的写接口
- 至少 1 份明确的请求体示例
- 至少 1 份成功返回示例
- 至少说明这些错误各自对应什么字段：
  - 不能为空
  - 上课时间缺失
  - 学生为空
  - 系统外学生姓名缺失
  - 系统外学生行政班缺失
  - 系统内老师 ID 非法
  - 系统外老师姓名缺失
  - 结束时间早于开始时间
  - 时间冲突
  - 点名时间未配置时是否允许创建
