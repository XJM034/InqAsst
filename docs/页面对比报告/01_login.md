# 页面对比报告：`/login`

## 1. 先说结论

`/login` 不是当前最严重的视觉偏移页。它的问题不在首页长得不像，而在于登录链路已经从 mock 的“本地角色映射”升级成真实验证码登录和校区选择流程，导致后续跳转和状态管理语义发生了变化。

换句话说，这一页属于“初始界面大体对齐，但交互链路已经真实化”的页面。

## 2. mock 页面想表达什么

mock 分支里的登录页非常明确，就是一个轻量登录壳：

- 输入手机号
- 输入验证码
- 点击登录
- 依据手机号直接映射到目标角色与首页

它的重点不是账号体系，而是“让人快速进入正确角色页面”，因此：

- 页面没有复杂的登录中间态
- 没有登录后的校区选择步骤
- 没有真实 token / session 语义
- 也没有接口失败后的复杂兜底状态

## 3. 当前接口版实际做成了什么

当前实现已经变成真实登录流程，主要代码在：

- `components/app/login-form.tsx`
- `lib/services/mobile-client.ts`

当前链路包含：

- `sendLoginCode`
- `loginWithCode`
- `selectCampusAfterLogin`

也就是说，登录页现在承担的不只是“进入系统”，而是：

- 发验证码
- 校验验证码
- 接收账号角色与可选校区
- 在部分场景下弹出校区选择
- 完成最终落地跳转

## 4. 页面展示差异

### 4.1 初始画面差异不大

- 初始仍是手机号输入框、验证码输入框、发送验证码、登录按钮。
- 因此用户第一眼看到的结构和 mock 基本接近。

### 4.2 交互层级比 mock 多了一层

- mock 点击登录后直接进入目标页。
- 当前接口版登录成功后，可能还要进入校区选择弹窗。

这会带来两个直接感受：

- 用户完成登录的路径更长了
- 登录页不再只是“表单页”，而是“认证起点页”

### 4.3 错误态和等待态更重

- mock 更多是演示态页面。
- 当前页必须处理发送验证码失败、验证码错误、校区选择失败、接口超时等情况。

因此当前页的状态密度天然比 mock 高。

## 5. mock 与实际接口数据差异

### 5.1 mock 数据语义

mock 本质上不是“登录数据”，而是“角色入口映射”：

- 手机号只是 key
- 输出是落地角色和目标页面

它更像 `resolveLoginDestination(phone)` 这种本地路由决策器。

### 5.2 当前接口数据语义

当前接口版的输入输出语义完全不同：

- 输入：手机号 + 验证码
- 输出：登录态、账号信息、校区范围、后续选校区动作

这里已经不是“演示性 mock 跳转”，而是真正的认证闭环。

### 5.3 这页为什么暂时不属于 P0 偏移页

因为它的页面目标并没有被错误替换：

- mock 是“登录”
- 当前还是“登录”

差别主要在于：

- 交互更真实
- 链路更长
- 状态更多

它没有像 `/admin/course-settings` 那样，从“课程规则配置页”变成“老师设置入口页”。

## 6. 对齐建议

- 保持首屏视觉结构继续贴近 mock，不要让真实登录状态把首屏改成另一个设计语言。
- 校区选择如果必须保留，视觉上要尽量当作登录后的子步骤，而不是重新做一层风格完全不同的页面。
- 登录成功后的落地逻辑要和 mock 角色预期一致，避免出现“能登进去，但去了不同首页”的体验割裂。
- 错误提示文案要尽量轻量，不要让真实接口状态把页面做得像后台管理表单。

## 7. 对齐时要重点核对的点

- 输入框顺序、占位文案、按钮文案是否保持 mock 风格。
- 发送验证码和登录按钮的禁用态是否影响版式。
- 校区选择是否只在必要时出现。
- 登录成功后的角色落地是否与 mock 预期路径一致。
