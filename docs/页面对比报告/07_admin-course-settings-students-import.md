# 页面对比报告：`/admin/course-settings/[courseId]/students/import`

## 1. 先说结论

这页在视觉上相对稳定，但它存在一个反向问题：看起来像接了真实接口，其实数据层目前仍主要停留在 mock 语义。

所以它不是严重视觉偏移页，却是“表面稳定、底层未完成”的典型页面。

## 2. mock 页面想表达什么

mock 中这页就是一个很明确的导入页：

- 告诉你支持哪些字段
- 给模板下载入口
- 允许上传学生名单

重点是让管理员快速完成批量导入，不需要太多复杂状态。

## 3. 当前接口版实际做成了什么

当前实现依然保持了相似的结构：

- 支持字段说明
- 模板下载
- 上传区域

因此它在视觉上没有像课程设置主页面那样发生大偏移。

## 4. 页面展示差异

### 4.1 初始视觉基本对齐

- 区块组织方式接近 mock
- 上传动作仍然清晰
- 模板下载入口仍保留

### 4.2 真正的问题不在展示层

这页最大的问题不是“长得不像”，而是“是否真的已经完成真实导入能力”。

## 5. mock 与实际接口数据差异

### 5.1 mock 数据来源

mock 使用 `adminCourseStudentImports`。

它已经足够支撑一页静态导入壳。

### 5.2 当前接口数据来源

当前 `getAdminCourseStudentImport` 仍主要返回 mock 式结构，而不是完整真实导入链路结果。

这说明：

- 页面外观能保持对齐
- 但能力层还没有真正完成

### 5.3 风险点

如果后续只看页面截图，会误以为这页已经对齐完成。

但实际上它至少还需要校验：

- 模板是否来自真实接口或真实静态资源
- 上传是否真的落到后端
- 导入结果提示是否真实

## 6. 对齐建议

- 保持当前视觉结构，不需要大改。
- 重点补齐真实导入链路，不要只保留 mock 壳。
- 页面状态要覆盖导入成功、部分失败、字段缺失、模板错误等真实场景。

## 7. 对齐时要重点核对的点

- 模板下载是不是可用真链路。
- 上传后是否真的触发导入，而不是本地假提交。
- 成功/失败反馈是否和 mock 的轻量文案风格保持一致。
