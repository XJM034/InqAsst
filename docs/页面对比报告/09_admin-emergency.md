# 页面对比报告：`/admin/emergency`

## 1. 先说结论

这页现在最大的偏差是“标题和列表还在，但语义从‘今日代课课程’偏成了‘本周老师设置总览’”。视觉上似乎没跑太远，实际上业务中心已经变了。

## 2. mock 页面想表达什么

mock 里的这页不是普通老师设置页，而是“今天有哪些代课课程要处理”。

它的重点是：

- 今天
- 代课
- 待处理课程

换句话说，这是一张“当日临时代课任务清单”。

## 3. 当前接口版实际做成了什么

当前实现主要在：

- `app/admin/emergency/page.tsx`
- `components/app/teacher-selection-client.tsx`
- `lib/services/mobile-app.ts` 中的 `getAdminEmergencyWeeklyData`

当前数据链路是：

- `fetchAdminTeacherSettingsOverview(getWeekStart())`

这里已经把数据视角切成了“周维度老师设置总览”。

## 4. 页面展示差异

### 4.1 外观还像，但标题重心变了

mock 的心理预期是“今日代课”。

当前页更像“老师设置”或“点名老师配置”。

标题一变，整页的任务感也变了。

### 4.2 featured 区域容易混入非代课课程

mock featured 应该突出今天真正要处理的课程。

当前如果从一整周老师设置总览里抽数据，就很容易把普通课程也带进来。

### 4.3 列表筛选维度偏成了周维度

mock 更偏“今日待处理”。

当前更偏“本周内按天浏览老师设置”。

这会让页面从异常处理入口，变成一般配置入口。

## 5. mock 与实际接口数据差异

### 5.1 mock 数据来源

mock 使用 `adminEmergencyData`。

它聚焦的是：

- 今日代课课程
- 今日待选老师
- 今日待处理状态

### 5.2 当前接口数据来源

当前使用：

- `getAdminEmergencyWeeklyData`
- `fetchAdminTeacherSettingsOverview(getWeekStart())`

它更擅长回答的是：

- 本周有哪些课程
- 各课节老师配置是什么

### 5.3 关键错位

mock 需要的是“今天真正异常的课程集合”。

当前拿到的是“本周老师设置总览”，两者不是一回事。

于是页面会自然产生这些偏差：

- featured 不够聚焦
- 列表里混进普通课程
- 页面重心落到配置，而不是落到代课处理

## 6. 对齐建议

- 恢复页面主标题和副标题中的“今日代课”语义。
- 列表数据要先过滤出今日待处理课程，再渲染页面。
- 如果继续保留按周切换，只能作为辅助浏览，不应该覆盖页面主语义。
- featured 区块必须只服务当天待处理课程，不要把普通课程放进去。

## 7. 对齐时要重点核对的点

- 用户打开页面时，是不是立刻知道“今天哪些课需要代课处理”。
- 页面里是否仍混入了本周普通课程。
- 从这里进入详情页时，是否感觉自己在处理“当天任务”，而不是维护“老师配置总表”。
