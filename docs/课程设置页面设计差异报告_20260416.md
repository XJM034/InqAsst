# 课程设置页面设计差异报告

## 1. 先说结论

现在的“课程设置”页面，并不是“原设计稿做得不够像”，而是**页面的业务含义已经变了**。

原设计稿里的这页，本质上是在做：

- 今天到底按哪一套课程规则行课
- 今天哪些课程算“生效课程”
- 哪些课程要临时加进来
- 哪些课程要临时移出去
- 保存后，老师端点名和管理端点名总控都按这份结果走

而现在页面实际做成了：

- 展示课程列表
- 查看老师相关信息
- 跳去“教师设置 / 临时代课设置”

所以问题不是单纯改几个颜色、改几个按钮文案就能解决，而是要把这页重新拉回“**当天课程规则配置页**”的定位。

---

## 2. 原设计稿这页到底是干什么的

原设计文件： [Attendance2.pen](</d:/proj/backproj/daoleme/Attendance2.pen>)

设计稿中，这一页的核心信息很明确：

### 2.1 页面上半部分

- 标题是“当天规则”
- 下面有一句说明：
  - 先决定今天按哪个规则行课，再进行搜索、增减和临时新增
- 有两个规则选项：
  - `默认（周二）`
  - `按照其他行课日行课`

这说明设计稿想表达的是：

- 今天的课，不一定完全按“今天星期几”的默认课表走
- 管理员可以临时决定：今天整体按另一套行课日来执行

### 2.2 页面中间部分

- 搜索框文案是：
  - `搜索课程名 / 上课地点`
- 列表标题是：
  - `今日生效课程`
- 右上角按钮是：
  - `临时新增课程`

这说明这部分不是普通“课程总表”，而是：

- 先圈定“今天到底有哪些课实际生效”
- 再对这个结果做搜索、补充、删减

### 2.3 每张课程卡片

设计稿里的课程卡片有这些信息：

- 课程名
- 状态标签：
  - `今日行课`
  - 或 `临时新增`
- 一行摘要信息：
  - 上课时间
  - 上课地点
  - 老师
  - 人数
- 两个操作按钮：
  - `进入学生名单`
  - `移出今日行课` 或 `保留到今日行课`

这说明设计稿想让管理员直接在这一页做“今日课程集合”的调整。

### 2.4 页面底部

- 有总结文案：
  - `今日最终生效 8 门课程，修改后老师端与点名总控使用同一份规则。`
- 有主按钮：
  - `保存课程设置`

这一句其实是整页最关键的信息，意思非常明确：

- 这页不是浏览页
- 这页是在改“今天真正生效的课程规则”
- 保存后，至少会影响：
  - 教师端看到的可点名课程
  - 管理端点名总控看到的课程

## 3. 现在页面实际上做成了什么

当前实现主要在：

- [app/admin/course-settings/page.tsx](/d:/proj/backproj/InqAsst/app/admin/course-settings/page.tsx:1)
- [components/app/admin-course-settings-client.tsx](/d:/proj/backproj/InqAsst/components/app/admin-course-settings-client.tsx:1)
- [lib/services/mobile-app.ts](/d:/proj/backproj/InqAsst/lib/services/mobile-app.ts:552)
- [lib/services/mobile-adapters.ts](/d:/proj/backproj/InqAsst/lib/services/mobile-adapters.ts:1055)

现在页面的数据是从“教师设置总览”拼出来的，不是从“今日课程设置”这个业务对象来的。

当前页面的实际行为大致是：

- 顶部写的是 `点名模式`
- 只有一个模式：`手动点名`
- 搜索框搜的是 `课程名称或老师`
- 列表标题是 `课程列表`
- 顶部操作按钮是 `临时代课设置`
- 每张卡片的角标显示的是“老师名字”
- 每张卡片第二个按钮是 `教师设置`
- 底部虽然有 `保存课程设置` 按钮，但当前前端并没有对应的整体保存逻辑

也就是说，现在这页更像：

- “课程列表 + 教师设置入口页”

而不是：

- “当天实际生效课程配置页”

## 4. 具体差异点整理

下面这部分可以直接理解成“为什么你会觉得现在和原设计差很大”。

### 4.1 页面定位已经变了

原设计：

- 管今天到底按什么规则上课
- 管今天最终生效哪些课程

现在实现：

- 主要围绕教师设置、临时代课设置展开

这不是样式差异，是**业务目标差异**。

### 4.2 顶部规则区完全不对

原设计要的是：

- `当天规则`
- 可以在“默认当天规则”和“按其他行课日行课”之间切换

现在实现是：

- `点名模式`
- 只有一个 `手动点名`

问题在于：

- `手动点名` 这个概念和设计稿中的“今天按哪套规则行课”不是一回事
- 现在这个区块没有表达“今天课程集合怎么得出来”

### 4.3 搜索维度不对

原设计：

- 搜索课程名 / 上课地点

现在实现：

- 搜索课程名称或老师

这说明当前页默认还是站在“查课程、查老师”的视角，不是“处理今天有效课程集合”的视角。

### 4.4 列表语义不对

原设计：

- `今日生效课程`

现在实现：

- `课程列表`

这两个名字差别很大。

`课程列表` 是静态总表。

`今日生效课程` 是当天经过规则筛选、允许人工干预后的结果集。

### 4.5 右上角主操作不对

原设计：

- `临时新增课程`

现在实现：

- `临时代课设置`

这两个动作完全不是一件事。

- `临时新增课程` 是往“今天生效课程集合”里加课
- `临时代课设置` 是改这节课由谁来点名

这说明当前页面已经把“课程设置”和“教师设置”混在了一起。

### 4.6 卡片右上角标签不对

原设计：

- `今日行课`
- `临时新增`

现在实现：

- 展示老师名，或者当前点名老师名

原设计标签想表达的是“这门课为什么出现在今天的有效列表里”。

现在标签表达的是“这门课是谁在带”。

表达的根本不是同一个维度。

### 4.7 卡片里的第二个按钮不对

原设计：

- `移出今日行课`
- `保留到今日行课`

现在实现：

- `教师设置`

这说明现在页面缺了最核心的动作：

- 把课程从“今日生效集合”里移除
- 把临时课程保留到今日结果里

### 4.8 卡片信息不完整

原设计卡片摘要包含：

- 时间
- 地点
- 老师
- 人数

现在页面实际主要能稳定拿到的只有：

- 课程名
- 地点
- 老师相关信息

其中“人数”现在没有直接在这个页面的数据链路里提供。

### 4.9 底部保存按钮名义上有，实际业务没闭环

原设计的保存含义是：

- 把今天的课程规则和最终生效课程一起保存下来

现在页面虽然有“保存课程设置”按钮，但当前页面数据结构本身并不是围绕这个业务来的，所以这个按钮没有承接原设计所要求的整页保存语义。

## 5. 为什么会变成这样

从当前代码看，页面数据不是从“课程设置”专门接口返回的，而是从教师设置接口拼出来的。

当前关键数据来源是：

- [lib/services/mobile-api.ts](/d:/proj/backproj/InqAsst/lib/services/mobile-api.ts:176) `fetchAdminTeacherSettingsOverview`

前端又在这里把它映射成课程设置页数据：

- [lib/services/mobile-adapters.ts](/d:/proj/backproj/InqAsst/lib/services/mobile-adapters.ts:1055)

这会带来一个直接后果：

- 因为底层拿到的是“教师设置总览”
- 所以前端只能很自然地渲染出“教师相关”的页面
- 但设计稿要的其实是“今日课程规则配置”

说白了：

- **现在不是前端把设计稿画歪了**
- **而是前端手里拿错了数据模型**

## 7. 现在后端覆盖不了设计稿上的哪些关键内容

这部分是最重要的。

按现在代码和已有文档来看，后端**没有完整覆盖**原设计稿这页真正需要的数据。

已有文档里也明确提到过这个问题：

- [docs/backend-integration-gap-report.md](/d:/proj/backproj/InqAsst/docs/backend-integration-gap-report.md:212)
- [docs/前后端联调冲突梳理_20260413.md](/d:/proj/backproj/InqAsst/docs/前后端联调冲突梳理_20260413.md:144)

核心缺口有这些。

### 7.1 缺“当天规则”这个对象

设计稿需要知道：

- 今天默认按星期几行课
- 当前管理员是不是切成了“按别的行课日行课”
- 如果切了，是按哪一天

现在没有一个明确的接口，直接告诉前端：

- 今天课程规则是什么
- 当前选中的规则是什么
- 可切换的规则有哪些

### 7.2 缺“今日生效课程列表”这个结果集

设计稿要展示的不是“所有课程”，而是“今天最终生效的课程”。

这里面至少可能包含三类来源：

- 今天按默认规则本来就应该上的课
- 因为切换行课规则后被带进来的课
- 临时新增进来的课

现在后端没有直接给出这样一个最终结果集。

### 7.3 缺“课程为什么出现在今天”这个状态字段

设计稿里的标签其实很关键，它在回答：

- 这是正常今日行课
- 还是临时新增

当前后端并没有给这个页面一套这样的状态字段。

### 7.4 缺“是否从今日生效集合移除/保留”的状态与动作

设计稿卡片上的按钮不是固定按钮，而是和课程当前状态有关：

- 正常今日行课的课，可以移出
- 临时新增的课，可以保留

当前后端没有给出这套针对“今日生效课程集合”的管理动作语义。

### 7.5 缺“今日最终生效课程数量”

设计稿底部要显示：

- 今日最终生效多少门课

现在前端也许可以自己用列表数量凑出来，但前提是得先拿到“最终生效课程列表”。

如果存在分页、筛选、权限过滤或者不同校区范围，最好后端直接返回总数。

### 7.6 缺“整页保存”对应的持久化语义

设计稿的“保存课程设置”不是保存单张卡片，而是保存：

- 今天按哪套规则行课
- 今天临时加了哪些课
- 今天移掉了哪些课
- 最终今天有效课程到底是什么

当前并没有看到一套与之对应的整页读写模型。

## 8. 要怎么改，才能改回原设计的样子

这一部分直接说建议。

## 8.1 第一步：先把页面定位改正

建议先统一结论：

- `/admin/course-settings` 这页应该回到“当天课程生效规则配置页”
- “教师设置 / 临时代课设置”不应该再占据这页主语义

更直白一点说：

- 这页要管“今天上哪些课”
- 不是管“这节课谁来代课”

教师设置应该仍然存在，但应作为独立链路，不应抢占这页主功能。

## 8.2 第二步：前端数据结构要重做

现在的 `AdminCourseSettingsData` 结构不够表达原设计。

建议把页面数据结构改成更接近下面这种意思：

```ts
type AdminCourseSettingsPageData = {
  title: string;
  rule: {
    title: string;
    description: string;
    currentMode: "default_day" | "other_teaching_day";
    defaultWeekdayLabel: string;
    selectedTeachingDay?: string;
    options: Array<{
      id: string;
      label: string;
      description: string;
      active: boolean;
    }>;
  };
  searchPlaceholder: string;
  sectionTitle: string;
  temporaryAddActionLabel: string;
  effectiveCourseSummary: string;
  effectiveCourseCount: number;
  courses: Array<{
    id: string;
    title: string;
    status: "today_active" | "temporary_added";
    statusLabel: string;
    timeText: string;
    locationText: string;
    teacherText: string;
    studentCount: number | null;
    rosterHref: string;
    action: {
      type: "remove_from_today" | "keep_for_today";
      label: string;
      enabled: boolean;
    };
  }>;
  saveButtonLabel: string;
};
```

重点不是字段名字，而是页面一定要能表达这几件事：

- 今天规则是什么
- 最终生效课程有哪些
- 每门课当前是什么来源状态
- 每门课能做什么调整
- 保存后会影响什么

## 8.3 第三步：后端最好新增一个“课程设置首页聚合接口”

这是最关键的建议。

不要再让前端用“教师设置总览接口”硬拼这页了。

建议单独做一个接口，例如：

```text
GET /api/admin/course-settings/overview
```

它专门返回这页所需的完整数据。

建议这个接口至少返回：

- 今天日期
- 今天默认星期几
- 当前实际采用的行课规则
- 可选规则项
- 今日最终生效课程列表
- 每门课的来源状态
- 每门课的时间、地点、老师、人数
- 页面底部汇总文案所需的统计信息

如果不做这个聚合接口，前端会继续面临这些问题：

- 一页要拼很多接口
- 规则和课程状态容易拼错
- 页面语义继续被教师设置数据带偏
- 保存时也很难知道该向哪里提交完整结果

## 8.4 第四步：补齐写操作接口

原设计不是只读页，所以读接口还不够，写接口也要补。

至少要考虑以下几类操作。

### 方案 A：分动作接口

适合后端按操作拆开做。

建议至少有：

- 切换当天规则
  - 例如：`POST /api/admin/course-settings/rule`
- 临时新增课程
  - 例如：`POST /api/admin/course-settings/temporary-courses`
- 移出今日行课
  - 例如：`POST /api/admin/course-settings/effective-courses/{id}/remove`
- 保留到今日行课
  - 例如：`POST /api/admin/course-settings/effective-courses/{id}/keep`

### 方案 B：整页草稿 + 保存接口

适合更贴近设计稿“最后统一保存”的感觉。

例如：

- `GET /api/admin/course-settings/overview`
- `PUT /api/admin/course-settings/overview`

前端把整页状态改完，一次性提交：

- 当前规则
- 临时新增课程列表
- 被移除课程列表
- 最终课程集合

如果让我建议，我更偏向：

- 读接口用聚合接口
- 写接口也尽量围绕“整页配置”建模

因为原设计最像“今天的课程执行方案配置页”，而不是若干彼此无关的小按钮。

## 8.5 第五步：把教师设置从这页拆回辅助链路

教师设置当然重要，但它应该是：

- 某一门课的附加设置
- 或独立页面链路

不应该覆盖“课程设置”首页的主语义。

建议做法：

- `/admin/course-settings` 专心做“今日课程规则配置”
- 教师设置继续留在：
  - `/admin/emergency`
  - `/admin/emergency/course/{courseId}`

如果确实希望两边有联动，也应当是：

- 课程设置决定今天哪些课生效
- 教师设置决定这些生效课程由谁点名

这才是合理关系。

## 9. 如果暂时不改后端，前端能不能先硬改出原样

可以改出“看起来更像”，但**不能真正改回设计稿的业务效果**。

原因很简单：

### 9.1 文案能改

可以把：

- `点名模式` 改成 `当天规则`
- `课程列表` 改成 `今日生效课程`
- `临时代课设置` 改成 `临时新增课程`

这些都能改。

### 9.2 样式能改

可以把页面排版、颜色、按钮、标签改得更接近设计稿。

### 9.3 但核心数据改不了

如果后端不补：

- 前端不知道今天到底按哪套规则行课
- 前端不知道哪些课是“今日生效”
- 前端不知道哪些课是“临时新增”
- 前端不知道“移出今日行课 / 保留到今日行课”要怎么落库

那最后只能做成：

- 视觉像了
- 业务还是假的

这个不建议。

## 10. 建议的改造顺序

为了尽量少返工，建议按下面顺序做。

### 第一阶段：先把业务定义说死

先统一这几个问题：

- “当天规则”到底允许怎么切
- “按照其他行课日行课”具体是按周几映射
- “临时新增课程”是新增已有课程的某次行课，还是新增一条临时课记录
- “移出今日行课”和“保留到今日行课”到底分别改哪张表、哪种状态
- 保存后老师端和管理端分别读取哪份结果

如果这一步不定，前后端都会反复返工。

### 第二阶段：后端先出首页聚合读接口

优先把 `/admin/course-settings` 这页真正需要的数据一次性吐出来。

先把页面读对，比先做按钮提交更重要。

### 第三阶段：前端把页面先还原成设计稿结构

这一步主要是：

- 规则卡片
- 搜索
- 今日生效课程列表
- 课程状态标签
- 底部汇总文案
- 保存按钮

先把页面“说正确”。

### 第四阶段：补写操作

再接：

- 切规则
- 临时新增
- 移出今日行课
- 保留到今日行课
- 保存

### 第五阶段：联动验证

最后一定要验证：

- 改完课程设置后，教师端当天可点名课程有没有变化
- 管理端点名总控看到的课程有没有同步变化
- 未到学生、班级点名这些依赖“今日课程集合”的页面有没有跟着变

这一步非常关键，因为设计稿已经明确说了：

- 老师端与点名总控使用同一份规则

## 11. 我给你的最终判断

如果目标是“改回原设计的样子”，那结论非常明确：

### 11.1 前端需要改

而且不是小修，是要把这页重新改回“当天课程规则配置页”的结构和交互。

### 11.2 后端大概率也需要改

因为当前后端并没有完整覆盖原设计所需要的页面模型。

最少也要补：

- 一个课程设置首页聚合读接口
- 一套围绕“当天规则 / 今日生效课程 / 临时新增 / 保存”的写接口或整页保存接口

### 11.3 现在最不建议的做法

最不建议的是：

- 不改后端
- 只改前端文案和样式
- 然后说“已经和设计稿差不多了”

因为这样只是看起来像，真正业务还是不对。
