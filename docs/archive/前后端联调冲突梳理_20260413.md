## 1. 我可以直接改的

### 1.1 教师首页 / 周课表请求漏传 `weekStart`

现状：

- 前端 `InqAsst/lib/services/mobile-api.ts` 里的 `fetchTeacherHome()` 直接请求 `/api/teacher/home`
- 同文件里的 `fetchCourseWeek()` 直接请求 `/api/courses/week`
- 两个请求都没有带 `weekStart`

后端现状：

- `daoleme/src/main/java/com/daoleme/api/course/controller/TeacherController.java`
  - `@GetMapping("/home")`
  - `@RequestParam LocalDate weekStart`
- `daoleme/src/main/java/com/daoleme/api/course/controller/CourseController.java`
  - `@GetMapping("/week")`
  - `@RequestParam LocalDate weekStart`
- `InqAsst/backend/API_FIELD_GLOSSARY.md` 和 `InqAsst/backend/swagger-local-offline.json` 也都把 `query.weekStart` 标成了必填

影响：

- 教师首页一旦走真接口，会直接因为缺少 `weekStart` 返回 400
- `/api/courses/week` 这个 helper 以后只要被启用，也会同样报错

建议：

- 前端统一按 `Asia/Shanghai` 计算当前周一
- 请求改成：
  - `/api/teacher/home?weekStart=YYYY-MM-DD`
  - `/api/courses/week?weekStart=YYYY-MM-DD`

这项是纯前端协议修正，我可以直接改。

### 1.2 对仍未联调完成的页面，先做“禁用/显式标注未接后端”也可以直接处理

现状：

- `InqAsst/lib/services/mobile-app.ts` 中以下数据源还直接返回 mock：
  - `getAdminCourseSettingsData()`
  - `getAdminCourseStudentForm()`
  - `getAdminCourseStudentImport()`
- 以下页面虽然有真实读取，但保存动作还没接后端：
  - `app/admin/emergency/course/[courseId]/page.tsx`
  - `app/admin/emergency/course/[courseId]/select-teacher/page.tsx`
  - `app/admin/emergency/course/[courseId]/external-teacher/page.tsx`

影响：

- 页面能打开，但容易让人误以为已经完成联调
- 现场测试时会把“页面可见”误判成“功能可用”

建议：

- 在未完成联调的入口上加显式状态：
  - “仅 mock”
  - “只读”
  - “保存未接后端”
- 或者临时隐藏保存按钮 / 改成禁用态

这项也是前端可直接落地的风险隔离措施。

## 2. 我需要你确认业务的

### 2.1 教师端是否允许直接标记“请假”

现状：

- 后端点名状态统一支持 `0=未到 / 1=请假 / 2=已到`
- 教师端 `components/app/attendance-session-client.tsx` 当前只做了“已到 <-> 未到”切换
- 管理端未到学生页倒是允许改成 `present / leave / absent / unmarked`

冲突点：

- 接口能力已经支持“请假”
- 但教师端 UI 现在没有暴露“请假”入口

需要确认：

- 教师端是否本来就不允许录请假，只允许录“到/未到”    回答：业务如此
- 还是教师端也应该支持三态点名 回答：只支持“到/未到”  

如果要支持三态，我可以直接改前端交互；如果教师端只允许两态，那现在的 UI 就是业务限制，不算接口冲突。

### 2.2 “更换点名老师”到底是永久改课程老师，还是临时改当天 / 本课节老师

现状：

- 前端页面文案是“老师设置 / 更换点名老师 / 恢复默认老师”
- 前端读取逻辑默认存在：
  - 一个默认老师 `PRIMARY`
  - 一个当前点名老师 `ROLL_CALL`
- 后端当前只有 `POST /api/admin/courses/{courseId}/teachers`
- 这个接口在 `AdminSettingsService.assignCourseTeacher()` 里只是新增一条 `course_teacher` 记录，不会删除旧的 `ROLL_CALL`

问题：

- 如果业务是“整个课程永久换点名老师”，当前接口也不够完整，因为没有覆盖旧 `ROLL_CALL`
只改临时当天
- 如果业务是“仅今天 / 仅本课节代课”，那当前模型更不够，已经不是简单前端问题
课节本来就是当天单课程的设定
需要确认：

- 老师设置页要改的是：
  - 课程级长期老师
  - 还是课节级 / 日期级临时代课老师

这个口径一旦定下来，后端和表结构怎么改才有方向。

### 2.3 课程设置首页里的“规则模式 / 临时调整 / 保存”具体控制什么

现状：

- `app/admin/course-settings/page.tsx` 页面是完整 UI
- 但 `getAdminCourseSettingsData()` 还完全走 mock
- 后端当前能提供的是：
  - 课程学生名单
  - 课程老师绑定
  - 点名时间设置
- 没有直接对应“规则模式/临时调整/保存整个课程设置页”的聚合接口或实体

需要确认：

- 这里的“规则模式”究竟是：
  - 课程老师规则
  - 点名规则
  - 名单规则
  - 还是一个纯前端筛选视图


不先定这个，前端很难正确拆接口。

### 2.4 学生新增 / 编辑页里的“学生ID”到底是什么

2026-04-20 已决议：

- 管理端页面展示的“学生ID”统一显示数据库 `studentId`
- `externalStudentId` 不再作为页面展示编号，也不再要求管理员在新增页输入
- 新增页拆成两条写链：
  - 当前校区已有学生：`GET /api/admin/campuses/{campusId}/students/search` + `POST /api/admin/courses/{courseId}/students/existing`
  - 当前校区没有该学生：`POST /api/admin/courses/{courseId}/students`
- 临时新增学生时，前端不再生成 `TEMP-*` 学生 ID；保存成功后由后端返回数据库 `studentId`

当前结论：

- 本轮不再把“学生ID”当作待确认的业务口径问题
- 页面与名单展示已经统一切换到 `studentId`
- 若未来产品要单独维护“学号 / studentCode”，应另起新需求，不回退当前方案

### 2.5 “外部老师”录入后应该是什么身份

现状：

- 页面 `app/admin/emergency/course/[courseId]/external-teacher/page.tsx` 想录入“姓名 + 手机号”
- 后端只有普通教师创建接口 `POST /api/admin/teachers`
- `TeacherCreateRequest` 只能建“当前校区教师”

需要确认：

- 外部老师录入后，是不是要： 允许不属于任何校区
  - 进入当前校区教师池
  - 只作为临时代课老师存在
  - 允许不属于任何校区

如果这块不确认，前端没法判断该走“建老师 + 绑定课程”，还是该走另一套临时老师模型。

## 3. 数据库字段需要调整的

### 3.1 `student` 表缺少“前端可维护的学生编号 / 学号”字段

2026-04-20 状态更新：

- 本轮已经不再把“页面学生ID”定义为待维护学号
- 当前管理端页面展示数据库 `studentId`，因此这条不再是本轮上线阻塞项
- `externalStudentId` 继续视为内部兼容字段，不对管理员暴露为正式“学生ID”

保留的长期观察：

- 如果后续产品确认需要独立“学号 / studentCode”字段，再单独评估是否改表
- 但这已经不是当前课程学生新增 / 编辑方案的前置条件

### 3.2 `course_teacher` 当前模型无法稳定支撑“唯一当前点名老师”

当前表结构：

- `course_teacher(course_id, teacher_id, role)`
- 唯一键是 `(course_id, teacher_id, role)`

问题：

- 这只能防止“同一个老师、同一个角色”重复插入
- 不能防止“一个课程存在多个 `ROLL_CALL` 老师”
- 但前端老师设置页默认假设“当前点名老师”是唯一的

当前后端行为：

- `AdminSettingsService.assignCourseTeacher()` 只追加，不覆盖旧角色

如果业务确认是“课程级唯一点名老师”，建议至少做下面其中一种：

- 在模型上保证一个课程只能有一个有效 `ROLL_CALL`
- 或者把 `course_teacher` 改成带有效态 / 生效区间的模型

如果业务确认是“按课节 / 日期临时老师”，那就不是加唯一约束这么简单了，而是要单独建课节级老师关系表。

课程级点名老师不唯一
