# 后端协同当前问题

更新日期：2026-04-29

## 1. 教师端点名补录临时学生

前端目标：老师在点名页发现学生不在当前课程名单里时，可以补录临时学生并随本次点名一起提交。

当前前端契约：

- `POST /api/courses/{courseId}/students?courseSessionId={courseSessionId}`
- 请求体按管理端临时学生口径提交 `name + homeroomClassId`
- 后端返回数据库 `studentId`

当前判断：

- 前端已按当前契约补齐 `courseSessionId`。
- shared dev 仍需后端确认教师侧写接口是否正确实现权限、课节、课程名单追加和返回值。
- 若接口返回 500 或泛化错误，不应继续按前端漏参处理。

## 2. 临时新增课程的空白课程创建口径

前端目标：管理员在 `/admin/course-settings/new-course` 直接按当天时间设置创建空白课程。

需要后端明确：

- 创建空白课程时是否直接使用当天“时间设置”里的上课时间与点名时间。
- 老师与学生的系统内 / 系统外来源字段。
- 至少 1 名学生的校验口径。
- 保存成功后如何进入“今日生效课程”。
- 字段缺失时返回可定位字段的错误，不返回泛化 `不能为null`。

## 3. 系统内代课老师冲突判断

已观察到的 shared dev 风险：

- 部分真实重叠老师会返回自己的冲突。
- 部分当天无课老师会被错误命中为固定老师冲突。
- 当前候选接口可能返回多个校区的系统内老师；如果业务不允许跨校区代课，需要后端明确过滤或错误口径。

当前要求：

- 后端需要复核冲突判断的老师、校区、日期、时间段和课节范围。
- 前端可以展示结构化冲突提示，但不能替代后端正确校验。

## 4. 合班 / 互换二选一

前端目标：管理员在单课“更换点名老师”流程里，当目标系统内老师同时间段已有课时，选择“合班”或“互换”。

当前前端契约：

- 合班：`PUT /api/admin/roll-call-teacher/merge-group`
- 互换：`POST /api/admin/roll-call-teacher/batch/options` + `PUT /api/admin/roll-call-teacher/batch`

需要后端保证：

- 合班和互换是或关系，不同时提交。
- 合班创建后老师端能按 `rollCallGroupId` 获取详情并提交。
- 管理端总控能按 `rollCallGroupId` 聚合展示。
- 互换需要最终态校验和原子提交，不能只完成前端入口。
- 失败时返回可区分的结构化错误，支持前端分别展示两条路径的失败态。

## 5. Shared Dev 稳定性

当前默认开发口径是本机前端 `next dev` + shared dev proxy。

仍需关注：

- 慢响应导致前端长时间等待。
- `/api/admin/emergency/weekly` 等聚合接口在部分数据范围下响应慢或间歇失败。
- 浏览器 direct 直打 shared dev 可能遇到 CORS，不作为当前默认验收口径。
- 需要区分 shared dev 抖动、前端超时保护、真实内测 / 线上问题。

## 6. Swagger / 离线文档与实际 controller 一致性

需要后端维护：

- 接口路径、query、body、返回字段与实际 controller 一致。
- 学生新增 / 编辑、临时学生、点名老师、合班 / 互换等写接口必须有明确字段约束。
- 权限、校区、老师类型、错误码需要和前端展示口径一致。

## 前端当前不再追加

- 不新增 `lib/mocks/`。
- 不启动本机 `daoleme` 作为默认联调前置。
- 不把 shared dev 结论直接写成内测 / 线上结论。
- 不把系统内代课、合班 / 互换写链宣称为已闭环，除非 shared dev 或真实环境完成复测。
