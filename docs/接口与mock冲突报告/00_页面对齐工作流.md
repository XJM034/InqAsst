# 页面对齐工作流

## 0. 配套文档
这份文档只负责“怎么改”。

同目录下还有两份必须配套使用的文档：

- `00_复核标准工作流.md`
  - 负责“怎么判通过 / 不通过”
- `00_推演事故防呆补充.md`
  - 负责“怎么防止推演先跑偏，再把代码和测试一起带偏”

固定顺序：

1. 先按本工作流快判、分类、实施
2. 只要页面存在写动作，就同步过一遍 `00_推演事故防呆补充.md`
3. 改完后，再按 `00_复核标准工作流.md` 给最终结论

不能只执行其中一份文档就宣布页面完成。

## 1. 目标
这份工作流只做三件事：

- 以 `d:\proj\backproj\teststaticPage` 和 `http://localhost:3001` 为页面基准
- 运行时只用真实接口，不允许 mock 参与页面运行
- 用尽量短的流程，尽快判断这页该改前端、前端聚合，还是必须先补后端

追求结果：

- 判断快
- 修改准
- 验收稳

## 2. 最高优先级
最高优先级不是“先改前端”，而是同时守住三件事：

- 页面结果和基准页一致
- 前后端真实链路联通
- 提交对象和页面业务对象一致

只要工作区里同时有前端和后端工程，就默认按“前后端联动任务”处理，而不是按“单前端页面任务”处理。

## 3. 不可放松的结果闸门
流程可以缩短，但下面 5 个闸门不能放松。

### 3.1 语义闸门
- 当前页面拿到的是正确业务对象，不是“字段像，但语义不对”的对象
- 语义修复只能作为业务补充，不能带偏基准页
- 如果语义修复和基准页冲突，默认先判定未完成

### 3.2 视觉闸门
- 页面结构、层级、主要卡片、文案位置与基准页一致

### 3.3 运行闸门
- 运行时只走真实接口
- 不允许 `mock fallback / mock 回退 / mock 临时兼容`
- 出错时优先保留正常骨架下的空数据态，不切成另一张页面

### 3.4 写动作闸门
只要页面存在下面任一动作，就必须额外过“提交对象正确”检查：

- 保存
- 提交
- 恢复默认
- 确认
- 切换规则
- 删除 / 移出
- 新增 / 录入

要求：

1. 明确页面到底在改哪个业务对象
2. 明确这次提交的生效范围
3. 明确页面上是否有这个对象或范围的表达
4. 明确用户是否显式选择了这个对象或范围
5. 明确提交参数里是否混入了“技术上拿得到、业务上没表达”的锚点字段

如果这 5 条没有答清楚，不允许先定提交入参，不允许先写 submit，不允许先补“链路能跑”的测试。

### 3.5 验证闸门
- 至少补齐必要测试
- 至少做一次与基准页的最终比对
- 交互页上的关键控件必须真的可交互，不能只做静态壳或伪交互

## 4. 第一阶段：15-30 分钟快判
每个页面开工前，不先改代码，先在 15-30 分钟内回答 4 个问题：

1. 基准页长什么样
2. 当前页偏在哪里
3. 当前页拿到的是不是对的业务对象
4. 这页属于哪一类

快判时只看这些东西：

1. `docs/页面对比报告/对应页面.md`
2. `docs/接口与mock冲突报告/对应页面.md`
3. `d:\proj\backproj\teststaticPage\app\...`
4. `d:\proj\backproj\teststaticPage\components\...`
5. `http://localhost:3001/对应页面`
6. 真实后端源码仓中的 `Controller / Service / DTO / Repository`
7. 本地联调启动方式、端口、入口

快判阶段只输出一个结论：

- `A. 纯页面层问题`
- `B. 真实接口够用，但需要前端聚合`
- `C. 必须先补后端接口或 DTO`

如果 30 分钟内还不能分到 A/B/C，说明问题没拆清，先不要进入实施阶段。

## 5. 开工前固定检查

### 5.1 前后端联动为默认起手式
如果当前工作区同时包含前端和后端工程，则每个页面开工时必须先串一遍完整链路：

1. 页面路由与基准页
   - `app/.../page.tsx`
   - `localhost:3001`
   - `teststaticPage`
2. 前端读链路
   - `mobile-app.ts`
   - `mobile-adapters.ts`
   - `mobile-schema.ts`
3. 前端写链路
   - `mobile-client.ts`
   - 页面按钮、表单、保存动作对应的提交函数
4. 后端真实链路
   - `Controller`
   - `Service`
   - `DTO`
   - `Repository / Query`
   - 实际写入口

这一步做完之前：

- 不能先下“后端没有”的结论
- 不能先把页面归类成纯前端问题
- 不能先改代码

### 5.2 开工前 5 分钟固定检查
每页开工前固定补做下面检查：

1. 先确认是否存在该页对应的独立报告文件
2. 先确认该页主要矛盾属于 `页面层 / 数据层 / 两者都有`
3. 先确认列表或卡片的组织维度是否与基准页一致
4. 先扫对应测试，确认是否还锁着旧语义
5. 先确认验证命令是否可能被权限或环境限制拦住
6. 先确认本页是否已经把“前后端联动链路”串清楚

### 5.3 中文文案与编码检查
中文文案问题不是“小样式问题”，而是页面可读性故障。

要求：

1. 先列清固定中文文案
2. 不允许把终端乱码复制回源码
3. 改中文文案时，测试文件必须同步检查
4. 收口前必须人工确认页面上中文可正常阅读
5. 优先收成正常 UTF-8 中文或稳定常量

### 5.4 编码污染禁止动作
直接禁止下面行为：

1. 把终端乱码复制回源码
2. 在已有正常中文页面里继续新增 `\uXXXX` 形式的中文文案
3. 只改页面不改测试，或只改测试不改页面真实文案
4. 把“转义中文也能显示”当成完成标准

## 6. 第二阶段：按页面类型走分支

### A. 纯页面层问题
适用条件：

- 真实接口语义对
- 前后端读写链路已核对
- 主要问题在 DOM、布局、文案位置、卡片顺序、样式层级

执行重点：

1. 直接对齐 `teststaticPage` 的页面结构和样式
2. 少改数据层
3. 不在页面里临时拼语义

### B. 前端聚合问题
适用条件：

- 真实接口已提供主要事实
- 缺的是前端 ViewModel，不是后端对象本身

执行重点：

1. 先定义页面最终 ViewModel
2. 优先改 `mobile-adapters.ts`
3. 再改 `mobile-app.ts`
4. 必要时补 `mobile-schema.ts`
5. 最后才改 `app/.../page.tsx`

禁止：

- 在页面组件里直接拼字段、写业务判断、临时硬兜文案

### C. 必须先补后端
适用条件：

- 缺的是对象本身，不是映射
- 当前真实接口没有这页必需语义
- 页面有真实写动作，但后端没有对应写入口或可落地字段

执行重点：

1. 先确认真实后端源码仓里确实没有现成接口或 DTO
2. 再补后端 `DTO / Service / Controller / Query`
3. 后端可用后，再回前端接线

禁止：

- 明明缺的是后端对象，却在 adapter 层硬拼
- 先做一版“看起来像”的前端假语义，再长期拖着不补后端

## 7. 实施阶段通用规则

### 7.1 页面层和数据层分开改
- 页面层负责结构、布局、视觉对齐
- 数据层负责接口、映射、聚合、空态、错误策略

### 7.2 页面只消费 ViewModel
推荐顺序：

1. `mobile-adapters.ts`
2. `mobile-app.ts`
3. `mobile-schema.ts`
4. `app/.../page.tsx`

补充约束：

- 语义修复只能落在 `schema / service / adapter / ViewModel`
- 页面层不能借“语义修复”之名擅自改标题、卡片结构、列表组织方式或主要文案
- 如果本轮触达中文文案，页面、共用组件、service 映射、测试断言要作为一组一起改

### 7.3 错误态优先收敛成正常骨架
- 页面骨架不变
- 动态值置空
- 用“暂无 / 待配置 / 未生效”承接真实空值

### 7.4 显式检查 SSR / Hydration 风险
首屏不要直接依赖：

- `window`
- `document.cookie`
- `localStorage`
- `Date.now()`
- `Math.random()`

### 7.5 页面一致与后端联通必须一起验
实施过程中不能出现下面顺序：

- 先把页面做得很像，最后才发现后端链路不通
- 先把数据接通，最后才发现页面结构已经偏离基准

正确顺序是边改边同时校验：

1. 这页现在长得像不像基准页
2. 这页现在读的是不是真实后端对象
3. 这页如果有写动作，提交是不是落到真实后端口子
4. 这页如果有写动作，提交对象是不是和页面业务对象一致

任何一项没确认，都不能提前报完成。

### 7.6 交互控件必须是真交互
只要页面属于下面任一类型，就不能只做“看起来像”的静态壳：

- picker
- 选择器
- 滚轮
- 搜索框
- 下拉列表
- 分页 / 加载更多
- 切换器

要求：

1. 关键控件必须真的能点、能滑、能选、能触发状态变化
2. 不能依赖透明原生控件伪装成自定义交互
3. 不能只用静态文本模拟“上一项 / 当前 / 下一项”
4. 不能把“确认按钮能回跳”误判成“交互已完成”

如果交互页只有页面壳，没有真实交互能力，就不能算链路通过。

## 8. 第三阶段：统一验收
每个页面结束前，统一过下面清单。

### 8.1 必过项
- 已与 `teststaticPage` / `localhost:3001` 做过对照
- 页面结构与主要视觉块已对齐
- 运行时只使用真实接口
- 没有 mock fallback
- 页面失败时仍保持正常骨架
- 没有明显 hydration 风险
- 固定中文文案渲染正常，没有乱码或编码串
- 如果页面存在写动作，已确认真实后端入口可落地
- 如果页面存在写动作，已确认提交对象与页面业务对象一致
- 已明确核对本页前后端联动链路
- 如果页面是交互页，已确认关键控件真实可交互，不是静态壳或伪交互

### 8.2 最少测试要求
至少补两类，推荐三类：

1. `service / adapter` 测试
2. `unit` 页面测试
3. `e2e` 对比测试

如果本轮改到固定中文文案，页面测试里至少覆盖一项：

- 标题
- section label
- button label
- placeholder
- 空态文案

如果本轮发现过乱码或编码污染，还要额外做一次同链路复扫：

- 页面文件
- 共用组件
- `service / adapter`
- `tests/unit`
- `tests/service`

### 8.3 结案标准
只有同时满足下面条件，页面才算完成：

- 语义对
- 页面像
- 运行真
- 后端通
- 提交对象准
- 测试过

只做到“语义更合理”或“差不多像”，都不算完成。

## 9. 一句话版本
先用 15-30 分钟把页面分成“纯页面层 / 前端聚合 / 必须补后端”三类，再按分支快速实施；只要页面有写动作，就必须先确认“页面到底在改谁、改多大范围、页面上有没有把这个对象说明白”，最后再用统一验收清单把质量锁住。
