# 推演事故防呆补充

## 0. 这份文档怎么用
这份文档不单独使用。

配套关系：

- `00_页面对齐工作流.md`
  - 负责“怎么实施”
- `00_复核标准工作流.md`
  - 负责“怎么判通过 / 不通过”
- 本文档
  - 负责“怎么防止推演先跑偏，再把接口、代码、测试一起带偏”

使用时机：

- 只要页面存在提交、保存、恢复默认、确认、切换规则、新增、删除等写动作，就必须同步过本文档

## 1. 这次事故的本质
这次不是“代码不会写”，而是“推演先跑偏，代码跟着跑偏”。

最典型的错误链路是：

1. 先接受了一个“看起来合理”的技术锚点
   - 例如 `courseSessionId`
2. 没先反问这个锚点是否真的存在于页面语义里
3. 没先确认它是不是当前业务对象的必要入参
4. 直接围绕这个锚点继续补链路、补测试、补文档
5. 最后形成“技术上能跑通，但业务上是脏逻辑”的实现

这类问题本质上叫：

- `实现推演先于业务对象确认`

## 2. 这次具体错在哪里
以 `14 /admin/time-settings/[detail]` 为例，错误不是一处，而是连续失守：

1. 把 `courseSessionId` 当成当然成立的提交锚点
   - 但页面上没有任何元素表达“当前课节”
   - 也没有任何交互让用户选择课节

2. 把“能从数据里拿到”误判成“业务上应该使用”
   - 前端确实能从接口里拿到 `courseSessionId`
   - 但“拿得到”不等于“这个页面应该用它做入参”

3. 把“最小改动”误判成“正确建模”
   - 为了不改页面，先保留了 `courseSessionId`
   - 这在技术上省事，但在语义上是脏的

4. 把“后端可反查”误判成“接口语义合理”
   - 后端能通过 `courseSessionId` 反查校区和日期
   - 但这只是实现手段，不是页面业务语义

5. 复核时没有单独检查“提交对象是否在页面可感知”
   - 只看了页面像不像、链路通不通、测试过不过
   - 没单独问：页面提交的这个对象，页面上真的存在吗

## 3. 哪些场景最容易复发
下面这些页面最容易再次发生同类事故：

1. 规则页 / 设置页
   - 页面表达的是规则
   - 实现里却偷偷用了实体主键

2. 聚合页
   - 页面看的是校区、日期、分组结果
   - 实现却拿单条记录做锚点

3. picker + detail 成对页面
   - picker 本来只该选值
   - 最后却被推演成 picker 直接提交

4. 页面上没有显式对象上下文的场景
   - 页面上没有课程名、没有课节、没有学生、没有老师对象卡片
   - 提交参数却塞进了这些对象的 ID

## 4. 必补的防呆规则
下面这些规则以后要当硬闸门，不是建议。

### 4.1 提交对象显式性检查
只要页面存在写动作，开工前必须先回答：

1. 这次提交改的是谁
2. 这个“谁”在页面上有没有直接表达
3. 用户有没有显式选择这个“谁”

如果这 3 个问题里有任意一个答不上来：

- 不允许先定接口入参
- 不允许先写 submit 逻辑
- 不允许先补“链路能跑”的测试

### 4.2 页面语义优先于实现锚点
如果页面语义是：

- 校区
- 日期
- 规则
- 分组
- 统一设置

那么接口入参必须优先围绕这些业务对象设计。

除非页面本身真的在表达该对象，否则禁止优先拿下面这些实现锚点顶上去：

- `courseSessionId`
- `studentId`
- `teacherId`
- 单条记录 ID

### 4.3 “拿得到”不等于“该使用”
只要某个字段满足下面任一情况：

- 前端 loader 里能拿到
- 后端能反查别的信息
- 用它改动更小

都不能直接当作采用理由。

必须再补一句：

- 它为什么和当前页面业务语义一致

如果补不出来，就默认不能用。

### 4.4 先写“业务提交句子”，再写接口
任何写动作，先写一句完整业务句子，例如：

- “保存上课时间 = 更新当前校区指定日期的全部上课时间”
- “保存点名时间 = 更新当前校区统一点名规则”

只有这句话能写顺，才允许继续定义：

- path
- body
- query
- DTO
- 前端 submit

如果业务句子里根本没出现某个参数，但接口里却出现了它，这就是红灯。

### 4.5 复核新增一项：提交对象与页面对象是否一致
以后复核时，除了：

- 页面一致
- 链路真实
- 测试通过

还必须单独过这一条：

- `提交对象是否与页面可见业务对象一致`

出现下面任一情况，直接判 `不通过`：

1. 页面没表达对象，但接口提交了对象 ID
2. 页面没选对象，但提交依赖对象 ID
3. 页面表达的是规则，接口却提交单实体主键
4. 页面表达的是聚合范围，接口却提交锚点记录

## 5. 新的执行顺序
以后遇到有写动作的页面，顺序固定改成：

1. 先写“页面在改什么业务对象”
2. 再写“提交生效范围是什么”
3. 再写“页面上是否有这个对象的直接表达”
4. 再定接口入参
5. 再定前端 submit
6. 最后才写实现

也就是：

- `先确认业务对象`
- `再确认生效范围`
- `最后才确认技术锚点`

不能反过来。

## 6. 最小防呆清单
以后只要页面有提交动作，开工前先过这 6 个问题：

1. 页面提交改的是哪个业务对象
2. 页面提交生效范围是什么
3. 页面上有没有这个对象的直接表达
4. 用户有没有显式选择这个对象或范围
5. 接口参数是否只包含业务句子里出现过的东西
6. 是否存在“只是为了方便反查而塞进去的主键参数”

第 6 条如果答案是“有”，默认先判为脏逻辑，除非能给出明确业务理由。

## 7. 一句话版本
以后先问“这个页面到底在改谁、改多大范围、页面上有没有把这个对象说清楚”，再去设计接口和实现；不能再先拿一个技术锚点把整条链路推出来。
