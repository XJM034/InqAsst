# 接口与 Mock 冲突：`/admin/control`

## 1. 冲突等级

`中高`

这页当前的核心冲突是：mock 关注“老师是否已经提交过点名”，而页面实现又混入了很多“考勤结果统计”语义，导致完成态和结果态容易被混在一起。

## 2. mock 依赖的对象

mock 总控页需要的是：

- 已完成多少班
- 未完成多少班
- 各班是否还待老师处理

这里的“已完成 / 未完成”口径应当是：

- `已完成`：老师已经提交过点名
- `未完成`：老师还没有提交过点名

也就是“流程推进对象”。

## 3. 当前接口实际使用的对象

当前链路是：

- `fetchAdminRollCallOverview`
- `fetchAdminTeacherSettingsOverview`
- `mapAdminControlData`

当前接口实际上同时提供两类信息：

### 3.1 完成态信息

- `rollCallCompleted`

它表示该课节是否已经有过点名提交记录，可直接用于判断：

- 已完成
- 未完成

### 3.2 结果统计信息

- `shouldAttend`
- `present`
- `leave`
- `absent`

这些更适合作为辅助展示，而不是首页主状态。

## 4. 具体冲突点

### 4.1 状态口径容易混淆

总控页第一层状态应当是：

- 已完成 / 未完成

其中：

- `已完成 = 老师已提交过点名`
- `未完成 = 老师尚未提交过点名`

不能把它误解成：

- 已到多少
- 是否全勤
- 是否还有缺勤/请假

### 4.2 结果统计会抢主语义

当前卡片上的：

- `x/y 已到`
- 进度条

本质上都属于结果统计或辅助信息。

如果页面用这些结果字段去反推“已完成 / 未完成”，就会把“老师已经提交过点名，但班上仍有缺勤/请假”的班级错误地看成未完成。

### 4.3 页面任务属性被削弱

mock 页面本来是给管理员看：

- 哪些班还没提交
- 哪些班已经提交
- 应该先催谁

如果首页主要强调“已到 / 请假 / 未到”，页面就会更像报表，而不是总控面板。

## 5. 处理建议

- `rollCallCompleted` 作为“已完成 / 未完成”的唯一主判定。
- `shouldAttend / present / leave / absent` 保留，但降级为辅助信息。
- 页面排序、计数、状态色都应优先服从“是否已提交过点名”。
- 如果后端未来不再直接给 `finishedCount / unfinishedCount`，前端也应从 `rollCallCompleted` 聚合，而不是从出勤结果倒推。
