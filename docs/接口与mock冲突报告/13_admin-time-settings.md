# 接口与 Mock 冲突：`/admin/time-settings`

## 1. 冲突等级

`极高`

这页和课程设置一样，是对象与粒度都错了的页面。mock 要规则对象，当前接口给的是课节列表对象。

## 2. mock 依赖的对象

mock 需要的是“时间规则首页对象”。

它的中心信息只有两类：

- `actualClassTime`
- `attendanceWindow`

并且这两个对象都带着：

- 周几
- 当前值
- 详情入口

也就是说，mock 首页需要的是“规则摘要对象”，不是课节数组。

## 3. 当前接口实际使用的对象

当前链路是：

- `getAdminTimeSettingsData`
- `fetchAdminTimeSettingsSessions`
- `mapAdminTimeSettingsData`

当前对象的中心是：

- `sessions[]`

也就是某天有多少课节，每个课节当前时间配置是什么。

## 4. 具体冲突点

### 4.1 对象冲突

mock 是：

- 规则对象

当前是：

- 课节列表对象

### 4.2 粒度冲突

mock 首页最小单元是：

- 一类规则

当前首页最小单元是：

- 一节课

### 4.3 路由冲突

mock 首页跳的是规则详情页。

当前首页实际跳的是课节详情页。

### 4.4 字段冲突

mock 要的是：

- 当前规则值
- 默认逻辑说明

当前给的是：

- 各课节当前配置结果

### 4.5 适配层冲突

只要 adapter 继续以 `sessions[]` 为中心输出首页数据，页面就不可能回到 mock 的两卡规则首页。

## 5. 当前接口能覆盖什么

- 展示某一天有哪些课节。
- 展示这些课节各自当前时间配置。
- 进入课节级明细。

## 6. 当前接口覆盖不了什么

- 把页面直接作为规则首页使用。
- 直接表达“当前这套实际上课时间规则是什么”。
- 直接表达“当前点名窗口规则是什么”。

## 7. 处理建议

- 这页必须重新抽象视图模型。
- 最好由后端给规则级对象。
- 若后端暂时做不到，前端也必须从多个课节真实返回结果中聚合出“规则摘要对象”，而不是把课节结果直接摆上首页。
- 任何聚合过程都不能混入 mock 字段或 mock 规则。

## 8. 当前页面备注

- 本页首页不保留搜索框。
- 原因：首页只保留两张固定规则卡片，搜索控件没有实际业务价值，反而容易在复核时把“是否可搜索”误判成页面缺陷。
- 后续复核这页时，不要再把“没有搜索框”判定为不通过项。
