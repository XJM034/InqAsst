# 接口与 Mock 冲突：`/admin/course-settings`

## 1. 冲突等级

`极高`

这是当前项目里最典型的“拿错对象导致页面整体变义”的页面。

## 2. mock 依赖的对象

mock `/admin/course-settings` 依赖的不是普通课程列表，而是“当天生效课程配置对象”。

这个对象至少应包含 5 组信息：

1. `dayRule`
   - 今天按哪套规则行课。
2. `effectiveCourses`
   - 今天最终生效的课程集合。
3. `courseStatus`
   - 每门课是默认生效、临时新增、还是被临时保留/移除。
4. `courseActions`
   - 支持“移出今日行课”“保留到今日行课”“进入学生名单”等动作。
5. `savePayload`
   - 整页统一保存时需要提交的结果。

也就是说，mock 页面需要的是一个“集合配置对象”。

## 3. 当前接口实际使用的对象

当前数据链路是：

- `getAdminCourseSettingsData`
- `fetchAdminTeacherSettingsOverview(getWeekStart())`
- `mapAdminCourseSettingsData`

这个对象本质上是：

- 某周老师设置总览
- 课程与老师配置关系
- 默认老师 / 临时代课老师信息

它天然关心的是“老师配置”，不是“今天课程集合”。

## 4. 具体冲突点

### 4.1 对象冲突

mock 需要：

- 当天课程规则对象

当前使用：

- 老师设置总览对象

这不是字段缺几项的问题，而是上游对象类型不对。

### 4.2 粒度冲突

mock 是：

- 今天整体规则
- 今天整体课程集合

当前对象是：

- 某周内每门课的老师配置摘要

它没有“今天这一整天最终结果”这个层级。

### 4.3 字段冲突

mock 需要的核心字段，当前对象里天然不存在：

- 今天按默认星期还是按其他行课日
- 今日最终生效课程集合
- 临时新增课程集合
- 临时移出课程集合
- 每门课为什么属于今天的状态说明

当前对象提供的却是：

- 老师是谁
- 是否有临时老师
- 与老师设置相关的课节信息

### 4.4 动作冲突

mock 的主动作是：

- 选规则
- 调整课程集合
- 统一保存

当前接口最适合驱动的动作却是：

- 查看老师
- 跳老师设置页
- 跳临时代课页

### 4.5 链路冲突

mock 中这页是课程配置主链路的起点。

当前接口版把它做成了老师设置入口页，导致：

- 首页 badge 跟着错
- 学生名单页失去“今日课程”上下文
- 教师端当天课程入口无法从这里得到稳定来源

## 5. 为什么会出现这种冲突

因为当前前端在真实接口里最容易拿到的是 `TeacherSettingsOverview`。

但这个对象再方便，也不等于 mock 需要的“今日课程配置对象”。

于是 adapter 发生了典型问题：

- 用老师配置对象去假装课程配置对象

结果是页面越接真实接口，越不像原页。

## 6. 当前接口能覆盖什么

当前接口还能覆盖这些内容：

- 展示课程基础信息
- 展示老师相关信息
- 判断是否存在临时代课配置
- 跳转到老师设置链路

## 7. 当前接口覆盖不了什么

当前接口无法自然覆盖：

- 当天规则切换
- 今日课程集合生成逻辑
- 课程集合新增/移出
- 整页统一保存语义
- “保存后老师端与总控共用同一份结果”的链路含义

## 8. 处理建议

### 8.1 结论先说

这页不应该继续在 adapter 层硬拼。

### 8.2 两种处理方向

方向 A：后端补“当天课程配置对象”

- 最理想。
- 直接给前端一包 `dayRule + effectiveCourses + savePayload`。

方向 B：前端聚合多接口真实返回值

- 如果后端暂时来不及补整包接口。
- 前端也必须显式聚合出“当天课程配置视图模型”，不能继续直接消费 `TeacherSettingsOverview`。
- 聚合来源也必须全部是现有真实接口，不允许混入 mock 数据。

### 8.3 明确禁止的做法

- 继续把老师设置数据伪装成课程设置数据。
- 继续让“老师设置”按钮占据课程集合动作位。
- 继续保留一个没有真实提交意义的“保存课程设置”壳按钮。
