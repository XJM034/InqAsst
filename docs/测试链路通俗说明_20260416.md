# InqAsst 测试链路通俗说明

日期：2026-04-17  
说明：这份文档按当前代码重写，明确参考了前端项目 `InqAsst` 和后端项目 `daoleme`，不再沿用早期 mock 时代的页面语义。

## 1. 先说现在系统是怎么跑的

现在这套系统的主链路已经非常明确了：

- 前端读链路基本都是：`页面 -> InqAsst/lib/services/mobile-app.ts -> InqAsst/lib/services/mobile-api.ts -> 后端接口`
- 前端写链路基本都是：`页面交互 -> InqAsst/lib/services/mobile-client.ts -> 后端写接口`
- 后端统一落在 `daoleme/src/main/java/com/daoleme/api/*/controller/*` 和对应的 `*Service`
- 当前真正的业务主键，已经从“页面自己猜今天第几节课”，切成了 `courseId + courseSessionId`

这意味着我们现在讲“测试链路”，不能再只说页面长什么样，而要说：

- 这个页面读的是哪几个真实接口
- 它最终依赖哪个后端 service
- 写回去以后，另外哪些页面会跟着变化

## 2. 这次重写后，最重要的几条新口径

和旧版说明相比，现在最关键的变化有 6 条：

1. 老师端首页的“代课”不再靠前端猜第二节课，而是看真实课节是否 `substitute`。  
   前端：`InqAsst/lib/services/mobile-adapters.ts` `mapTeacherHomeData`  
   后端：`CourseReadService.resolveTeacherVisibleSessions`、`TeacherTodaySessionResponse.substitute`

2. 老师端首页和点名页，登录后不再静默回落到 mock 数据。  
   前端：`InqAsst/lib/services/mobile-app.ts` `getTeacherHomeData`、`getAttendanceSession`

3. 老师点名页和老师名册页都围绕 `courseId + courseSessionId` 读真实课节，不再是旧的 demo 固定数据。  
   前端：`app/teacher/attendance/demo/page.tsx`、`app/teacher/home/roster/page.tsx`

4. 时间设置已经改成：  
   - 同校区同一天的“实际上课时间”一起改  
   - “点名时间”走校区统一规则，不再支持单课节单独改点名窗  
   后端：`CourseSessionTimeSettingService`

5. 所有读时间的页面，都统一经过 `CourseSessionTimeResolver` 读取“当天生效时间”，不再混用原始课节时间和覆盖时间。  
   后端：`CourseSessionTimeResolver`

6. 老师端和管理端对点名状态的权限不同：  
   - 老师端只能改“已到 / 未到”，请假只展示，未点名进入页面时按已到展示  
   - 管理端能展示全部状态，但只能改成“已到 / 请假 / 未到”，未点名是展示态  
   前端：`attendance-session-client.tsx`、`admin-class-attendance-client.tsx`  
   前端映射：`mobile-adapters.ts` `mapAttendanceSession`

## 3. 统一看法：读链路、写链路、鉴权

### 3.1 读链路怎么走

当前页面读真实数据时，通常是这样：

- 页面组件发起加载
- 调 `InqAsst/lib/services/mobile-app.ts`
- `mobile-app.ts` 再调 `InqAsst/lib/services/mobile-api.ts`
- `mobile-api.ts` 用 `serverRequestJson` 把 `inqasst_token` 带到后端
- 后端 controller 收请求
- 后端 service 查数据库并组装 DTO

关键代码：

- 前端带 token：`InqAsst/lib/services/http-server.ts`
- 前端读接口封装：`InqAsst/lib/services/mobile-api.ts`
- 前端页面级装配：`InqAsst/lib/services/mobile-app.ts`

### 3.2 写链路怎么走

当前页面提交数据时，通常是这样：

- 页面点击按钮或修改状态
- 调 `InqAsst/lib/services/mobile-client.ts`
- 直接请求后端写接口
- 后端写数据库
- 前端再跳回上一页或 `router.refresh()`
- 其他页面重新走读接口，所以会看到最新结果

关键代码：

- 前端写接口封装：`InqAsst/lib/services/mobile-client.ts`
- 后端点名写入：`AttendanceApplicationService.submit`
- 后端老师设置写入：`CourseSessionRollCallTeacherService`
- 后端时间设置写入：`CourseSessionTimeSettingService`、`AdminSettingsService`

### 3.3 登录和鉴权怎么走

前端：

- 页面：`InqAsst/app/login/page.tsx`
- 交互：`InqAsst/components/app/login-form.tsx`
- 客户端接口：`InqAsst/lib/services/mobile-client.ts`
- cookie：`InqAsst/lib/services/auth-session.ts`

后端：

- 登录接口：`daoleme/.../user/controller/UserController.java`
- 登录逻辑：`daoleme/.../user/service/UserService.java`
- 当前用户资料：`daoleme/.../user/controller/ProfileController.java`

当前真实规则：

- `/` 直接跳 `/login`
- `/role-select` 现在只是兼容路由，也直接跳 `/login`
- 一个手机号如果同时命中管理员和老师，后端按管理员处理
- 一个手机号如果命中多个管理员校区，后端返回 `selectionRequired`，前端弹出校区选择
- 登录成功后，前端会写 `inqasst_token`、角色、校区等 cookie

## 4. 老师端主链路

## 4.1 老师首页 `/teacher/home`

前端页面：

- `InqAsst/app/teacher/home/page.tsx`
- `InqAsst/components/app/teacher-home-client.tsx`

前端读链路：

- `getTeacherHomeData`
- `fetchMeProfile`
- `fetchTeacherHome`
- `fetchTeacherTodaySessions`
- `mapTeacherHomeData`

后端接口：

- `GET /api/me`
- `GET /api/teacher/home`
- `GET /api/teacher/courses/today`

后端实现：

- `ProfileController.me`
- `TeacherController.home`
- `TeacherController.today`
- `CourseReadService.teacherHome`
- `CourseReadService.listTodayForTeacher`
- `CourseReadService.resolveTeacherVisibleSessions`

当前真实业务口径：

- 老师能看到的课，既包括课程默认老师的课，也包括被临时指派成点名老师的课
- 如果课程默认老师不是当前老师，但当前老师成了这节课的实际老师，这节课就算“代课”
- 老师首页现在要求同时展示“主课模块 + 代课模块”；只有一条代课数据时，也允许同一条数据同时渲染成两种形态
- 主课模块不再额外挂“代课”标签；代课语义只在专门的代课模块上体现
- “明日行程”只能取上海时区的“明天”，不能退化成“下一次有课”

测试时应该重点看：

- 当前账号今天有没有课
- 今天的主课和代课是否都出现在首页
- `明日行程` 是否只反映明天，不串到周内其他天
- 当前页面是否真的在吃 live 数据，而不是 mock

## 4.2 老师点名入口 `/teacher/attendance`

前端页面：

- `InqAsst/app/teacher/attendance/page.tsx`

前端读链路：

- 先调 `getTeacherHomeData`
- 再从 `home.daySchedules` 里只找“上海时区今天”那一天
- 如果今天没课，跳 `/teacher/attendance/no-class`
- 如果今天有课，跳到对应 `actionHref`

后端依赖：

- 仍然是老师首页那条读链路：`/api/me` + `/api/teacher/home` + `/api/teacher/courses/today`

当前真实业务口径：

- 点名入口不能复用首页的 `defaultDayKey`
- 首页允许今天没课时默认选到本周其他有课日展示
- 点名入口不允许这么做，它只能判断“今天能不能点名”

测试时应该重点看：

- 今天没课时是否稳定去 `/teacher/attendance/no-class`
- 今天有课但不在点名时间时，是否进入只读名单而不是正式点名
- 不要把未来课节当成今天课节

## 4.3 老师点名页 `/teacher/attendance/demo`

前端页面：

- `InqAsst/app/teacher/attendance/demo/page.tsx`
- `InqAsst/components/app/attendance-session-client.tsx`

前端读链路：

- `getAttendanceSession(courseId, courseSessionId)`
- 组合读取：
  - `/api/me`
  - `/api/teacher/home`
  - `/api/courses/{courseId}`
  - `/api/courses/{courseId}/students?courseSessionId=...`
  - `/api/courses/{courseId}/attendance/latest?courseSessionId=...`

前端写链路：

- `submitTeacherAttendance`
- `POST /api/courses/{courseId}/attendance`

后端实现：

- 读：`CourseController.detail`、`students`、`latestAttendance`
- 写：`CourseController.submitAttendance`
- 业务：`AttendanceApplicationService.submit`
- 权限：`CourseAccessService.assertCanView / assertCanSubmitAttendance`

当前真实业务口径：

- 老师端提交点名时，必须带 `courseSessionId`
- 提交内容必须覆盖整节课全部应到学生，后端会校验“不能漏人，也不能多传”
- 老师只能在点名窗口内提交
- 点名窗口来自“校区规则 + 实际上课时间”
- 老师端页面里：
  - 只能改 `已到 / 未到`
  - `请假` 只展示不可点
  - `未点名` 进入老师端后默认按 `已到` 展示
- 页面在点名窗口开启时，会轮询 `/api/courses/{courseId}/attendance/status`，如果后端记录变了且本地没改动，会自动刷新

测试时应该重点看：

- 正常提交是否成功
- 不在点名时间时是否被后端拦住
- 老师端是否真的只有两态可切
- 提交后是否回老师首页

## 5. 管理端主链路

## 5.1 管理首页 `/admin/home`

前端页面：

- `InqAsst/app/admin/home/page.tsx`

前端读链路：

- `getAdminHomeData` -> `/api/admin/home/summary`

后端：

- `AdminController.homeSummary`

当前真实业务口径：

测试时应该重点看：

- 首页规则卡时间是否和当天生效时间一致
- 今日代课统计是否和老师设置页面一致

## 5.4 未到学生页 `/admin/unarrived`

前端页面：

- `InqAsst/app/admin/unarrived/page.tsx`
- `InqAsst/components/app/admin-unarrived-client.tsx`

前端读链路：

- `getAdminUnarrivedData`
- `/api/admin/absent-students`

后端：

- `AdminController.absentStudents`
- `AdminReadService.absentStudents`

当前真实业务口径：

- 这页按“异常学生”出发，不是按课程出发
- 后端会把今天各课节的最新点名结果、花名册、当前负责老师一起拼成行
- `已到` 学生不会出现在这里
- `未点名 / 请假 / 未到` 都可能出现

- 补充：教师端只能改状态为已到、未到；请假状态可以显示；未点名状态的学生显示成已到

测试时应该重点看：

- 老师刚提交未到后，这里能否出现
- 管理端修正为已到后，这里能否消失
- 搜索是否按学生名 / 行政班 / 课程名 / 老师名一起生效

## 5.5 课程设置与学生名单 `/admin/course-settings`

前端页面：

- 列表：`InqAsst/app/admin/course-settings/page.tsx`
- 名单：`InqAsst/app/admin/course-settings/[courseId]/students/client.tsx`
- 新增/编辑：对应 `new`、`edit` 页面

前端读链路：

- 列表：`getAdminCourseSettingsData` -> `/api/admin/course-settings/overview`
- 名单：`getAdminCourseRoster`
  - `/api/courses/{courseId}`
  - `/api/admin/courses/{courseId}/students?courseSessionId=...`
  - `/api/admin/course-settings/overview`

前端写链路：

- 新增学生：
  - 页面不再预加载全量学生目录
  - 输入至少 2 个字后，通过 `GET /api/admin/campuses/{campusId}/students/search?q=&limit=10` 轻量搜索当前校区已有学生
  - 命中已有学生时，自动带出 `studentId / name / homeroomClassId`
  - 选择已有学生后，通过 `addExistingStudentToCourse` -> `POST /api/admin/courses/{courseId}/students/existing`
  - 未命中时，按临时学生处理；前端不再生成临时 `externalStudentId`
  - 临时学生最终通过 `createStudent` -> `POST /api/admin/courses/{courseId}/students`，只提交 `name + homeroomClassId`，由后端返回数据库 `studentId`
- 编辑学生：`updateStudent` -> `PUT /api/admin/courses/{courseId}/students/{studentId}`

后端：

- 概览：`AdminController.courseSettingsOverview` -> `AdminReadService.courseSettingsOverview`
- 学生 CRUD：`AdminController.createCourseStudent`、`updateCourseStudent`
- 业务：`AdminSettingsService`

当前真实业务口径：

- 课程设置首页展示的是“今日生效课程”，不是旧 mock 的静态课程列表
- 学生名单页已经是“课节维度名单页”，所以要尽量保留 `courseSessionId`
- 前端不再自己重排学生，名单顺序基本沿用后端 `/students` 返回顺序

测试时应该重点看：

- 从课程设置进名单页时，`courseSessionId` 是否带对
- 新增学生页输入姓名时，是否能正确出现本校区已有学生匹配下拉
- 点击已有学生后，姓名 / 学生ID / 行政班是否自动带出
- 未选择已有学生时，是否仍能只填姓名 + 行政班，并在确认后成功按临时学生保存
- 新增、编辑完是否回到正确名单页
- 名单接口与名单页里显示的“学生ID”是否都是数据库 `studentId`

测试时应该重点看：

## 5.7 时间设置 `/admin/time-settings`

前端页面：

- 列表：`InqAsst/app/admin/time-settings/page.tsx`
- 详情：`app/admin/time-settings/[courseSessionId]/client.tsx`
- picker：`app/admin/time-settings/[courseSessionId]/picker/client.tsx`

注意：

- 但当前页面实际把路径段当作 `class-time` / `attendance-window` 两个设置键来用
- 真正读写时，不是按这个路径段去找后端单课节接口

- 实际上课时间：`updateAdminCampusActualTime` / `resetAdminCampusActualTime`
- 点名规则：`updateAdminCampusRollCallRule`

后端：

- 读列表：`AdminController.sessionTimeSettings` -> `CourseSessionTimeSettingService.listForDate`
- 写实际上课时间：`PUT/DELETE /api/admin/campuses/{campusId}/actual-time`
- 写点名规则：`PUT /api/admin/campuses/{campusId}/roll-call-rule`
- 生效时间解析：`CourseSessionTimeResolver`

当前真实业务口径：

- 同校区同一天的“实际上课时间”是一套统一值
- 改一条，实际上是改当天该校区全部课节
- 单课节点名时间 override 已经被后端禁用
- 列表和详情里看到的点名时间，都是按“校区规则 + 实际上课开始时间”推导出来的

测试时应该重点看：

- 改完实际上课时间后，老师首页、总控、未到学生页是否都变成同一套时间
- 改完点名规则后，老师端可提交时间窗是否跟着变
- 不要再把周四设置串到周五

## 6. 三条最重要的跨角色联动

## 6.1 老师点名 -> 管理端看到 -> 管理端修正

这条链路现在的真实顺序是：

1. 老师在 `/teacher/attendance/demo` 提交  
2. 后端 `AttendanceApplicationService.submit` 写 `attendance_record` 和 `attendance_record_item`  
3. 管理端 `/admin/control` 通过 `/api/admin/roll-call/overview` 看到  
4. 管理端 `/admin/unarrived` 通过 `/api/admin/absent-students` 看到  
5. 管理端 `/admin/control/[classId]` 还能继续修正  

补充，在老师点名时间内，管理端间隔10s轮询查看学生状态

这是目前最核心的一条业务闭环。

## 6.2 管理端改点名老师 -> 老师端看到代课

这条链路现在的真实顺序是：

1. 管理端在 `/admin/emergency/...` 改 `courseSessionId` 的当前老师  
2. 后端 `CourseSessionRollCallTeacherService` 落库  
3. 老师端首页读取 `/api/teacher/home` + `/api/teacher/courses/today`  
4. 后端 `CourseReadService.resolveTeacherVisibleSessions` 把这节课算进新老师名下  
5. 前端 `mapTeacherHomeData` 把它映射成代课模块  

这里的关键不是“管理员保存成功”，而是“老师端是否真的改了”。


## 6.3 管理端改时间 -> 老师端显示和权限一起变化

这条链路现在的真实顺序是：

1. 管理端改实际上课时间或点名规则  
2. 后端把时间规则落到当天同校区  
3. `CourseSessionTimeResolver` 影响老师首页、总控、未到学生页的展示时间  
4. `AttendanceApplicationService.assertWithinAttendanceWindow` 影响老师和管理员能不能提交点名  

也就是说，时间设置不是纯展示配置，它会直接影响能不能写数据。

## 7. 当前必须记住的业务规则清单

下面这些是现在写测试时必须带着的规则：

1. 老师端点名一定要带 `courseSessionId`。
2. 老师端只能改 `已到 / 未到`，`请假` 只展示，`未点名` 进老师页时按 `已到` 展示。
3. 管理端能展示全部状态，但只能改成 `已到 / 请假 / 未到`。
4. 代课的定义不是“第二节课”，而是“这节课的实际负责老师不是默认老师”。
5. 老师首页必须允许“主课模块 + 代课模块”同时出现；必要时可由同一条课节数据渲染成两种形态。
6. “明日行程”必须只看上海时区的明天。
7. 同校区同一天的实际上课时间一致。
8. 点名时间不再按单课节单独配置，统一来自校区规则。
9. 所有展示课节时间的地方，都应该读“生效时间”，不能有页面还在读原始 `course_session.start_at/end_at`。
10. 兼容路由 `/role-select`、`/teacher/attendance/demo/roster` 不是主入口，测试不能把它们当成新的业务主页面。

## 8. 建议现在怎么测

如果按当前系统去做验证，顺序建议是：

1. 先测登录分流：老师登录、管理员登录、多校区管理员选校区。
2. 再测老师主链：老师首页 -> 点名入口 -> 点名页 / 名册页。
3. 再测跨角色主链：老师点名 -> 管理总控/未到/班级详情同步。
4. 再测管理员写链：班级点名修改、老师设置、时间设置、学生新增编辑。
5. 最后测联动：老师设置改完后老师端是否变；时间设置改完后老师端时间和可提交窗口是否变。

时间设置测试的顺序很重要，因为要修改上课、点名时间到当前时间，点名流程才能跑起来

不要再按旧思路把页面分成“纯 demo 页”和“真实页”。  
当前代码里，老师端和管理端大多数核心页面都已经围绕真实接口和真实 `courseSessionId` 运转了。

## 9. 一句话总结

现在这套系统最重要的不是“某个页面能不能打开”，而是：

- 老师端是不是按真实课节读数据
- 管理端是不是按真实课节写数据
- 时间和老师调整之后，跨页面、跨角色能不能一致

这份文档后面如果还要继续更新，也应该继续按这个方式写：  
直接写“页面 -> 前端 loader -> 后端接口 -> 后端 service -> 关键业务规则”，不要再回到旧 mock 语义。
